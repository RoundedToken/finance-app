# Coach — нуджи по качеству данных (AI-трек, шаг 1)

## 1. Context & Problem

Вся достоверность системы держится на ручном вводе: не записал траты — балансы
дрейфуют; не обновил снапшот — net worth врёт; ведро без baseline — баланс неточен.
Эти проблемы **тихие**: открывая дашборд за цифрами, ты не замечаешь, что цифры уже
протухли. Owner-решение (2026-06-27): тихий помощник, который раз в день смотрит на
финансы и **пишет в Telegram только при реальном сигнале** (молчит, когда всё ок).

Это **шаг 1 AI-трека** (roadmap 2.4): детерминированные правила в Worker `scheduled()`
(БЕЗ LLM), переиспользуют существующий cron + бот. Цель — улучшать **качество данных**
и ловить пороговые события, а не пересказывать то, что видно на дашборде. Умные инсайты
(аномалии в натуральном языке, тренды) — шаг 2 через Claude Code routine, по триггеру
«доказано, что нуджи читаются».

## 2. Goals

- G1: Раз в день Worker (`scheduled()`, отдельный утренний cron) собирает финкартину,
  прогоняет детерминированные правила и **при наличии сигналов** шлёт ОДНО Telegram-
  сообщение владельцу через бота (`sendMessage` на `telegram_id` = `chat_id` приватного чата).
- G2: **Только когда есть что сказать** — нет сигналов → нет сообщения. Никакого daily-digest.
- G3: **Cooldown** (таблица `coach_state`): один и тот же сигнал не повторяется чаще раза
  в `COOLDOWN_DAYS` (дефолт 4) — чтобы не долбить тем же.
- G4: Правила (все на уже доступных данных / `getDashboard`):
  - **Качество данных** (ядро): гэп без трат; устаревшие снапшоты; ведро без baseline;
    позиции без курса (net worth недосчитан).
  - **Бюджеты/цели**: категория-бюджет в статусе «over» (SPEC-020, факт >100% месячного лимита —
    консервативно, без pace-prorate); цель просрочена (`deadline < today`). ETA-отставание «ещё в
    сроке, но темп не дотягивает» — descope в routine (шаг 2): требует проекции по вкладам.
  - **Net worth / runway** (порог, НЕ сводка): свободные деньги < 0 (цели недообеспечены);
    runway ниже пола.
  - **Аномалия трат** (грубая, шаг 1): категория за этот месяц ≥ 2× трейлинг-среднего и выше
    абсолютного пола. (Тонкая версия — шаг 2 routine.)
- G5: Приоритизация + лимит: качество данных → бюджеты/цели → net worth → аномалии; в одно
  сообщение не больше `MAX_SIGNALS` (дефолт 4), остальное ждёт следующего дня/cooldown.
- G6: Чистый модуль `coach.ts` (правила — тестируемые функции на D1-mock); тонкий хук в
  `scheduled()`; изоляция (падение нуджа не валит rates-cron).

## 3. Non-Goals

- NG1: LLM / Claude Code routine (шаг 2). Здесь только детерминированные правила в Worker.
- NG2: Free-form чат «спроси про деньги» (3.1) — по триггеру, позже.
- NG3: Daily/weekly digest «всё хорошо» (осознанный non-goal roadmap — золочение для того, кто
  смотрит дашборд). Сообщение — только при сигнале.
- NG4: Net worth/runway как ежедневная сводка. Только пороговые сигналы (free<0, runway<пол),
  не «твой net worth такой-то».
- NG5: Change-based тренды («упало на 15%») — нужна история/нюанс, это шаг 2 (routine).
- NG6: Мульти-канальная доставка (Mini App push), coach_profile/опросы — non-goals roadmap.
- NG7: Настройки порогов в UI. Пороги — константы в коде (тюнятся правкой + деплой); UI —
  отдельно, если понадобится.

## 4. User journeys

### Happy path — есть сигнал
1. Утром cron будит Worker `runDailyNudge(env)`.
2. Собирается финкартина (`getDashboard` + last-expense/last-snapshot/per-bucket-baseline +
   per-category суммы).
3. Правила дают список сигналов; cooldown-фильтр убирает недавно отправленные.
4. Осталось ≥1 → компонуется ОДНО сообщение (топ-`MAX_SIGNALS`) → `sendMessage` владельцу.
5. Отправленные ключи пишутся в `coach_state` (last_fired = сегодня).

### Happy path — всё ок
1. Cron, сбор, правила → 0 сигналов (или все на cooldown) → **сообщение не отправляется**.

### Edge cases
- E1: Нет авторизованных юзеров / нет `telegram_id` → ничего не шлём (лог).
- E2: `getDashboard` бросил/пусто (нет данных) → нудж тихо не падает, rates-cron не затронут (G6).
- E3: Сигнал держится (данные всё ещё протухли) → повтор не чаще `COOLDOWN_DAYS` (нагинаем мягко).
- E4: Несколько сигналов одного типа (3 ведра без baseline) → агрегируем в одну строку, один ключ.
- E5: Пустая система (только начал вести) — `denom`/средние малы; правила с порогами не палят ложно
  (gap считается от последней траты; при 0 трат вообще — не палим gap, палим «нет данных» мягко/никак).
- E6: Telegram `sendMessage` упал (5xx) → лог, `coach_state` НЕ обновляем (чтобы повторить завтра).

## 5. Data model

### Миграция 0018 — `coach_state` (cooldown)

```sql
CREATE TABLE IF NOT EXISTS coach_state (
    signal_key  TEXT PRIMARY KEY,   -- 'gap_no_expenses' | 'budget_over' | 'bucket_no_baseline' | ...
    last_fired  TEXT NOT NULL,      -- YYYY-MM-DD последней отправки
    last_detail TEXT                -- опц. снимок значения (для будущей change-детекции)
);
```

Семантика: один ключ на тип сигнала (агрегированный). Cooldown = `today − last_fired ≥ COOLDOWN_DAYS`.
Никаких новых полей в существующих таблицах. `schema.sql` обновляется параллельно.

## 6. API contract

**Без HTTP-эндпоинтов.** Триггер — Cloudflare Cron (`scheduled`), не запрос. Доставка — Telegram
Bot API `sendMessage` (уже используется ботом). Auth наружу — не добавляется.

- `wrangler.toml`: добавляется cron `"0 7 * * *"` (07:00 UTC = ~09:00 CET, утро) для нуджа;
  существующий `"0 */6 * * *"` (rates) сохраняется. `scheduled(event)` ветвится по `event.cron`.

## 7. UI / UX

UI нет. UX — Telegram-сообщение, HTML, лаконичное:

```
🔔 Финансы — на заметку

🔸 Нет трат 4 дня — записал всё?
🔸 Снапшоты не обновлялись 16 дней — балансы дрейфуют.
🔸 Бюджет «Кафе» превышен на 22% (месяц на 2/3).
🔸 Свободные деньги ушли в минус — цели недообеспечены.
```

Тон — короткий, по делу, на русском. Каждая строка — один сигнал + что сделать.

## 8. Security

- Доставка — только авторизованным (`authorized_users`), через существующий бот-токен. Бот молчит
  для прочих (ADR-009) — не нарушается.
- Новых auth-поверхностей / эндпоинтов нет. `TELEGRAM_BOT_TOKEN` уже секрет.
- В логи — только счётчики/ключи сигналов, не суммы (PII не расширяем).
- `scheduled` доступен только Cloudflare (env внутренний).

## 9. Acceptance criteria

- [x] AC1: Отдельный cron `"0 7 * * *"` будит нудж; `scheduled` ветвится по `event.cron` (rates-cron
      не затронут, оба работают изолированно).
- [x] AC2: Правила качества данных срабатывают на синтетике (D1-mock): гэп без трат (>GAP_DAYS),
      устаревшие снапшоты (>SNAPSHOT_STALE_DAYS), ведро с событиями без baseline, missing_rates>0.
- [x] AC3: Бюджет в статусе «over» (SPEC-020) и цель просрочена (`deadline < today`) → сигналы.
      (ETA-отставание в сроке — descope в routine, шаг 2.)
- [x] AC4: free<0 → сигнал; runway<RUNWAY_FLOOR → сигнал; иначе net-worth молчит (не сводка).
- [x] AC5: Категория ≥SPIKE_FACTOR×трейлинг-среднего и ≥SPIKE_FLOOR_EUR → сигнал аномалии.
- [x] AC6: 0 сигналов → `sendMessage` НЕ вызывается (молчание).
- [x] AC7: Cooldown: сигнал, отправленный <COOLDOWN_DAYS назад, не повторяется; ≥COOLDOWN — снова.
- [x] AC8: Приоритизация + лимит MAX_SIGNALS; отправленные ключи записаны в `coach_state`;
      при ошибке `sendMessage` — `coach_state` не обновлён (E6).
- [x] AC9: Падение нуджа не валит rates-cron (try/catch изоляция); пустая/ошибочная картина → тихо.
- [x] AC10: `tsc --noEmit` + `npm test` зелёные; новые unit-тесты на правила + cooldown (D1-mock).

## 10. Test plan

- **Worker vitest** (D1-mock `node:sqlite`): для каждого правила — позитив/негатив на синтетике
  (seed expenses/snapshots/budgets/goals/rates); cooldown (coach_state с last_fired сегодня → молчит,
  старый → палит); приоритизация/лимит; «0 сигналов → не шлём» (мок sendMessage-счётчик); изоляция
  (нудж бросил → rates-cron ок). `signalsFor(...)` — чистая функция, основной объект тестов.
- **Regression**: 181 прежних; rates-cron поведение не изменилось.
- **Прод-smoke**: после деплоя — ручной триггер (или дождаться cron), проверить, что при реальном
  сигнале приходит сообщение, при отсутствии — нет. (Backup D1 перед миграцией 0018, правило 7.)
- **Review**: `solution-architect` (изоляция/чистота правил/типы) + `senior-qa` (матрица правил, cooldown,
  edge E1-E6, что молчит когда надо).

## 11. Risks & open questions

- R1: Пороги (GAP_DAYS=3, SNAPSHOT_STALE_DAYS=14, RUNWAY_FLOOR=3 мес, SPIKE_FACTOR=2, SPIKE_FLOOR=50€,
      COOLDOWN_DAYS=4, MAX_SIGNALS=4) — стартовые, тюнятся по факту (правка констант + деплой). Калибровка
      по реальному поведению (R-после-наблюдения).
- R2: Аномалия трат детерминированно груба (ложные срабатывания на сезонных/разовых) → высокий порог +
      cooldown; тонкая версия — шаг 2 routine (NG1).
- R3: `chat_id == telegram_id` для приватного чата — верно для 1-on-1 с ботом (наш кейс). Если появятся
      группы — пересмотреть.
- R4: «Шумность» — главный риск продукта (привыкнешь игнорить). Митигация: молчание-по-умолчанию (G2),
      cooldown (G3), лимит (G5), фокус на actionable качества данных. Калибруем по реакции.

## 12. Out of scope для review

- LLM/routine (шаг 2, NG1), чат (NG2), UI-настройки порогов (NG7), change-based тренды (NG5) — отложены.
- Грубость аномалии трат (R2) — сознательно (тонкая версия = routine).
