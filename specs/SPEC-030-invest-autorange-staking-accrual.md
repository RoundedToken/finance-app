# Инвестиции — авто-окно графика + живой прогноз стейкинга

> Фидбэк owner (2026-06-09) после SPEC-029: (1) «большая часть графика просто пустая — надо по
> умолчанию показывать за то количество времени, сколько есть данных»; (2) «стейкинг как будто не
> работает — пару дней застейкано, а пишется +0». Owner-решение по стейкингу: **живой прогноз
> накопления** по APR (растёт ежедневно) + ожидаемый €/год; факт из снапшота — отдельно.

## 1. Context & Problem

**График.** SPEC-029 дал выбор периода, но дефолт — 12 мес. У новой позиции (покупки 06-05, снапшот
06-07) 11.5 месяцев — пустые нули: спарклайн плоский и неинформативный. Месячная гранулярность для
короткой истории даёт 1–2 точки.

**Стейкинг.** «Доход стейкинга · факт» = прирост qty над покупками, берётся из снапшота баланса
(ground truth, ADR-018). Снапшот 06-07 ≈ сумме покупок → прироста 0 → факт €0. За 2 дня и объективно
≈ €0.19 (0.929 ETH × 2.61% × 2/365). Прогноз APR есть только пунктиром на графике — числом «сколько
накапало» не видно, выглядит как «не работает».

## 2. Goals

- **G1** (авто-окно): дефолт графика инвестиций — окно **от первой операции позиции** до сегодня
  (а не фикс. 12 мес). Пресет «Авто» — дефолтный; ручные пресеты (12м/6м/год/всё/период) остаются.
- **G2** (гранулярность): `value_series` адаптивна по длине окна — короткая история (дни/недели) →
  частые точки (≈дневные), длинная → реже (до ~40–50 точек), последняя точка = today.
- **G3** (живой стейкинг): в карточке — **прогноз накопления** дохода по APR с даты старта (растёт
  ежедневно) + **ожидаемый €/год**; **фактический** доход (из снапшота) — отдельной строкой.

## 3. Non-Goals

- **NG1**: смена семантики факта дохода стейкинга — остаётся из снапшотов (ground truth, ADR-018);
  прогноз — отдельная, явно помеченная «ожидаемо» величина, НЕ в net worth.
- **NG2**: новое поле `staked_since` в БД — дату старта аппроксимируем (last_snapshot_date / первая
  покупка); отдельное поле — только если потребуется точность (open question).
- **NG3**: крупный line-chart портфеля (NG1 SPEC-029) — по-прежнему вне scope.
- **NG4**: смена гранулярности на дашборде — там остаётся помесячно (только инвестиции адаптивны).

## 4. User journeys

### Happy path (график)
1. Stepan открывает «Инвестиции» → дефолт «Авто»: график за [первая операция .. сегодня].
2. История 4 дня → ~4 дневные точки (видна динамика), не 11 пустых месяцев.
3. Переключает на «12 мес»/«Всё»/«Период» → окно и гранулярность подстраиваются.

### Happy path (стейкинг)
1. Карточка ETH: «Доход стейкинга — прогноз накопления **+€0.19** (с 07.06, 2.61% APR) · ожид. **€35/год**».
2. Отдельная строка: «факт (из снапшота 07.06): +€0.00». Подсказка обновить баланс — для факта.
3. Каждый день прогноз растёт → видно, что стейкинг «капает».

### Edge cases
- **E1**: нет операций у позиции → авто-окно fallback на 12 мес (как раньше).
- **E2**: позиция не застейкана (staked_qty=0) → прогноз/ожидаемый не показываем.
- **E3**: APR неизвестен (нет override и Lido недоступен) → прогноз не считаем (показываем «—»/факт).
- **E4**: custom-период короче истории → точки только в окне; гранулярность по длине окна.
- **E5**: очень длинная история («Всё», годы) → точек не больше cap (≈50), без переполнения.

## 5. Data model

Изменений в D1 нет. `value_series` и прогноз стейкинга — by-read линза (`investments.ts`).
Дата старта стейкинга аппроксимируется `last_snapshot_date ?? первая покупка` (NG2).

## 6. API contract

### `GET /v1/web/investments` (расширение ответа)
- Auth/query — как SPEC-029 (`from/to/today` опц.). Без `from` → **авто-окно от первой операции**.
- `positions[].value_series` — адаптивная гранулярность (поле `date` — конкретная дата точки).
- Новые поля позиции:
  - `staking_forecast_eur: number | null` — накопленный прогноз дохода по APR с `staked_since` до today.
  - `staking_expected_annual_eur: number | null` — ожидаемый годовой доход (stakedValue × apr).
  - `staked_since: string | null` — дата отсчёта прогноза (для подписи).
  - (существующие `staking_income_eur/_qty` — факт из снапшота — без изменений.)

Внутри: `getInvestments` определяет `earliest` (min первой tx/snapshot) per bucket; `seriesStart =
from ?? earliest`; точки с шагом `step = max(1, ceil(days/45))` дней, последняя = `seriesTo`.

## 7. UI / UX

`InvestmentsPage`:
- `PeriodPresets` получает параметр `presets`; для инвестиций — набор с «Авто» (дефолт). `presetRange("auto")
  = {}` (без from/to → серверное авто-окно).
- `PositionCard`, блок стейкинга:
  - «Доход стейкинга — прогноз: **+€Z** (с DATE, APR%) · ожид. **€W/год**» (прогноз помечен как
    ожидаемый, не реальные деньги — в тон пунктиру forecast, memory dashed-line-means-forecast).
  - «факт (снапшот DATE): +€X» — отдельной строкой; подсказка обновить баланс для факта.

## 8. Security

- Только чтение; `from/to` валидируются (ISO), series in-memory. Auth — admin JWT. Новых поверхностей нет.
- Прогнозные величины явно помечены «ожидаемо», чтобы не путать с реальными деньгами/net worth.

## 9. Acceptance criteria

- [x] AC1: дефолт графика инвестиций — окно от первой операции позиции (не 12 мес). ✅ тест + визуал «Авто».
- [x] AC2: короткая история → частые (дневные) точки; длинная — реже (≤46), последняя = today. ✅ тесты (5 точек/4 дня; cap).
- [x] AC3: пресет «Авто» — дефолт; ручные пресеты работают как в SPEC-029. ✅ визуал.

- [x] AC5: `staking_expected_annual_eur` = stakedValue × apr; незастейканная/без APR → null. ✅ unit.
- [x] AC6: факт дохода стейкинга (из снапшота) показан отдельно и не изменился. ✅ визуал (строка «факт») + регресс.
- [x] AC7: дашборд (общий `PeriodPresets`) не затронут — набор пресетов параметризован, «Авто» не протёк. ✅ визуал дашборда.

## 10. Test plan

- **Worker (vitest)**: `investments.test.ts` — авто-окно (earliest→today, длина/границы), адаптивный шаг
  (короткое vs длинное окно, cap), прогноз стейкинга (формула накопления, expected annual, null-кейсы).
- **Regression**: весь `vitest run` + typecheck (worker + admin); факт дохода стейкинга не съехал.
- **Admin SPA**: Playwright — инвестиции с короткой историей (дневные точки) + блок прогноза стейкинга
  light/dark; дашборд не затронут.

## 11. Risks & open questions

- **R1**: аппроксимация даты старта стейкинга (`last_snapshot_date ?? первая покупка`) может быть неточной,
  если стейк начат позже снапшота. Митигация: достаточно для «живого» индикатора; точное `staked_since`
  поле — OQ1.
- **R2**: пресет «Авто» в общем `PeriodPresets` не должен протечь на дашборд. Митигация: набор пресетов —
  параметр компонента; дашборд использует набор без «Авто».
- **OQ1**: нужно ли явное поле `staked_since` в `investment_settings` для точного отсчёта? (пока нет).
