# Адаптивные бюджеты (RBAR)

> **Статус разработки:** только спецификация + план. Реализация — позже (см. `docs/adaptive-budgets-plan.md`).
> Эта спека описывает **полный** механизм; внедрение строго фазовое, advisory-first.

## 1. Context & Problem

SPEC-020 дал статичные **ручные** месячные лимиты по категориям. Но лимит, выставленный один раз руками, быстро устаревает: траты дрейфуют (тренд по тотал +6%/мес на нашей истории), категории живут в разных режимах, а «правильную» планку приходится переставлять вручную. Нужен **детерминированный механизм** (не AI/ML), который по истории трат сам предлагает реалистичный лимит на следующий месяц: поджимает при устойчивой экономии, держит при разовом переборе, откатывает при сильном, разрешает базе расти в периоды роста уровня жизни — и масштабирует осторожность по волатильности категории.

Исследование (2026-05-31, 8 семейств алгоритмов, верификация на 29 мес реальных данных) выбрало механизм **RBAR — Robust Baseline + Asymmetric Ratchet** и вскрыло критичную ловушку: наивная асимметрия «легко поднимать лимит на перебор» **дрейфует лимит вверх +23–35% из чистого шума и анти-экономит**. RBAR это исключает anti-drift-инвариантом. Подробности и источники — `[[adaptive-budget-mechanism]]` (memory) и `docs/adaptive-budgets-plan.md`.

## 2. Goals

- **G1**: Каждая расходная категория **автоматически классифицируется** в архетип из её истории: `fixed | recurring | seasonal | lumpy | intermittent | cold-start`. Классификация показывается пользователю (он может переопределить).
- **G2**: Для **recurring/seasonal** категорий система считает **рекомендованный лимит** `L_{t+1}` на следующий месяц по закону RBAR (damped-Holt база + asymmetric ratchet + slew + turbulence-gain + floor) и показывает его рядом с текущим ручным лимитом, **с человекочитаемой причиной** («3 мес экономии → поджать до €491»).
- **G3**: Для **lumpy** категорий (Техника/Одежда/Дом) система показывает **накопительный годовой конверт** (sinking fund): сколько «накоплено / доступно потратить», вместо бессмысленного месячного лимита.
- **G4**: Все суммы — **информационная линза** поверх реальных денег (решение owner'а 2026-05-31): никакого второго баланса, который можно «потратить ещё раз». Профицит/накопления = подсказки, не отдельный кошелёк. Сходится с инвариантом `free = net − targeted` (SPEC-013).
- **G5**: **Advisory-first** (решение owner'а): v1 **ничего не меняет автоматически**. Применение рекомендации = явный клик пользователя (переиспользует `PUT /v1/web/budgets/:id` из SPEC-020). Авто-применение — отдельная поздняя фаза после валидации.
- **G6**: Состояние оценщика **не хранится** — рекомендация детерминированно **переигрывается из полной истории трат** при каждом запросе (идемпотентно, устойчиво к до-вводу задним числом; в духе «балансы не хранятся», SPEC-011).
- **G7**: Закон управления — **единый чистый модуль** (`rbar.ts`), тестируемый без D1, переиспользуемый Admin-эндпоинтом, bootstrap'ом Mini App (lens) и будущим cron-нуджем (roadmap 2.4).
- **G8**: **Бэктест на исторических 29 мес** — обязательный acceptance: показать траекторию рекомендаций по каждой recurring-категории и **доказать отсутствие апдрейфа** (главный риск из исследования).

## 3. Non-Goals

- **NG1**: Не AI/ML. Только детерминированные алгоритмы (robust статистика, damped-Holt, SPC/CUSUM, slew-rate). Воспроизводимо, white-box.
- **NG2**: Не авто-применение в v1. Лимиты двигает только пользователь (G5). Авто-режим — Фаза 3 плана, отдельным owner-решением после бэктеста.
- **NG3**: Не отдельный «кошелёк»/баланс для конверта или профицита (G4). Только линза-подсказка. Маршрутизация профицита в `goals` — возможная будущая итерация, **не** в этой.
- **NG4**: Не оптимизируем параметры закона по пользовательской истории (на `N=12–29` это переобучение). Параметры — фиксированная глобальная политика по архетипу (см. §5 константы).
- **NG5**: Не трогаем `fixed` (Жильё) авто-сжатием — это needs. Только change-point алерт «аренда изменилась, проверь лимит».
- **NG6**: Не делаем push/нотификации (это roadmap 2.4, отдельным проектированием). v1 — визуальная рекомендация в Admin + read-only lens в Mini App.
- **NG7**: Не делаем CRUD адаптивных настроек в Mini App. Mini App — **только read-only lens** (остаток/конверт), как уточнено в CLAUDE.md §11. Управление (override архетипа/пола, accept) — Web Admin.
- **NG8**: Не вводим savings-first / 50-30-20 глобальный потолок в этой итерации (нужен доход + savings-target в данных) — отложено в Фазу 4 плана.

## 4. User journeys

### Happy path (Admin, advisory)
1. Открываю **/budgets**. Кроме статичного прогресса (SPEC-020) вижу **бейдж архетипа** у каждой категории (`recurring`, `lumpy`, …) и, где применимо, **карточку рекомендации**: «Текущий лимит €520 → рекомендуем **€491** ↓ · причина: *3 мес в рамках бюджета, база снижается*».
2. Жму **«Применить»** на рекомендации → лимит категории обновляется (под капотом `PUT /budgets/:id` с рекомендованным значением). Toast «Лимит обновлён». Или **«Скрыть»** — рекомендация прячется на этот месяц.
3. У **lumpy** категории (Техника) вместо месячной рекомендации вижу **конверт**: «Годовой бюджет ≈ €1130 · отчисление €94/мес · **накоплено €620** · доступно потратить сейчас».
4. У **fixed** (Жильё) рекомендации нет, но если сработал change-point — баннер «Похоже, аренда изменилась (€650→€720). Обновить лимит?».
5. (Onboarding/верификация) Раздел **«Как система видит мои категории»**: таблица архетипов с метриками (CoV, % нулевых, тренд). Могу **переопределить архетип** или задать **ручной пол** категории, если авто-классификация неверна.

### Happy path (Mini App, lens)
6. Ввожу трату: на плитке категории с бюджетом — read-only остаток (SPEC-020). Для **lumpy** категории вместо месячного остатка — **«в конверте Техника ≈ €620»**. Помогает решить «тянет ли конверт эту покупку», не открывая Admin.

### Edge cases
- **E1** (cold-start): у категории `< 6` месяцев данных → архетип `cold-start`, рекомендации нет, показываем «собираю данные (N/6 мес), лимит пока ручной».
- **E2** (классификация на границе): zero_frac ≈ 0.40 (Здоровье/Подарки) → гистерезис 15% не даёт «прыгать» между lumpy↔recurring при до-вводе. Архетип меняется только при уходе метрики за порог на >15%.
- **E3** (разовый всплеск): месяц €606 в Еде (база ~€470) → winsorize ±3·σ_MAD не пускает всплеск в базу; рекомендация **не** обваливается.
- **E4** (перебор после экономии): факт > лимита, но рост базы не подтверждён 3 мес → рекомендация = **HOLD** (не поджать и не поднять). Причина: «разовый перебор, держим планку».
- **E5** (сильный перебор): факт > лимита·1.25 → рекомендация = откат к `L_comfort` (последняя комфортная планка). Причина: «сильное превышение, возврат к комфортной планке».
- **E6** (подтверждённый рост): база растёт ≥3 мес (рост дохода) → рекомендация растёт шагом ≤g·5%/мес. Причина: «устойчивый рост 3 мес, поднимаем планку».
- **E7** (трата без курса на дату, `toEurAt`=null): пропускается в сумме месяца, инкрементит `missing_rates` (как SPEC-020); если месяц критично неполный — рекомендация помечается `low_confidence`, не показывается как «применить».
- **E8** (seasonal без 2 циклов): `< 24 мес` → seasonal-категория работает как recurring с широким коридором; сезонный профиль и YoY-CUSUM включатся позже (предупреждаем в reason).
- **E9** (категория деактивирована): не классифицируется, рекомендаций нет (JOIN по `is_active=1`, как SPEC-020 E5).

## 5. Data model

**Ключевой принцип:** состояние оценщика (`B_lvl, B_trd, σ_R, streak*, L_comfort`) **НЕ хранится** — переигрывается из `expenses` при каждом запросе (G6). Поэтому новые таблицы минимальны: только **пользовательские override'ы** и **аудит-лог принятых рекомендаций**. Миграция **`0012_adaptive_budgets.sql`** (`schema.sql` обновляется параллельно — правило 2). Существующие таблицы не меняются.

```sql
-- 0012_adaptive_budgets.sql — Фаза 2 пост-MVP (SPEC-023): адаптивные бюджеты RBAR
--
-- Добавляет:
--   - budget_settings: per-category override'ы (архетип, ручной пол, вкл/выкл адаптации).
--                      Состояние оценщика НЕ хранится — переигрывается из expenses.
--   - budget_recommendation_log: аудит-лог принятых/скрытых рекомендаций (для будущего
--                      авто-режима и объяснимости). Append-only.

CREATE TABLE IF NOT EXISTS budget_settings (
    category_id        TEXT PRIMARY KEY REFERENCES categories(id),
    archetype_override TEXT CHECK (archetype_override IN
                          ('fixed','recurring','seasonal','lumpy','intermittent')),  -- NULL = авто
    floor_eur          REAL CHECK (floor_eur >= 0),     -- ручной абсолютный пол лимита; NULL = авто (0.6·median)
    adaptive_enabled   INTEGER NOT NULL DEFAULT 1,      -- 1 = считать рекомендации; 0 = только ручной лимит SPEC-020
    created_at         TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at         TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS budget_recommendation_log (
    id              TEXT PRIMARY KEY,                    -- UUID
    category_id     TEXT NOT NULL REFERENCES categories(id),
    period          TEXT NOT NULL,                       -- "YYYY-MM" месяц, НА который рекомендация
    archetype       TEXT NOT NULL,
    prev_limit_eur  REAL,                                -- лимит до (может быть NULL — лимита не было)
    reco_limit_eur  REAL NOT NULL,
    reason_code     TEXT NOT NULL,                       -- SAVINGS_STREAK | TRACKING_DOWN | TRACKING_UP |
                                                         -- HOLD_AFTER_BREACH | ROLLBACK | FLOOR_HIT | COLD_START
    decision        TEXT NOT NULL DEFAULT 'pending'
                       CHECK (decision IN ('pending','accepted','dismissed')),
    decided_at      TEXT,
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_reco_log_cat_period
    ON budget_recommendation_log(category_id, period);
```

**Семантика и константы политики** (фиксированы в коде `rbar.ts`, не в БД, NG4 — не оптимизируются):

| Константа | Значение | Смысл |
|---|---|---|
| `ALPHA` (Holt level) | 0.25 | вес последнего winsorized месяца в базе |
| `BETA` (Holt trend) | 0.08 | скорость обновления наклона |
| `PHI` (damping) | 0.9 | гашение тренда (асимптота, не «вечный +6%») |
| `WINSOR_C` (Hampel) | 3.0 | порог винзоризации всплеска в σ_MAD до EWMA |
| `K_MAD` | 1.4826 | согласованность MAD↔σ |
| `STEP_DOWN` | 0.025 | макс. шаг лимита вниз за месяц |
| `STEP_UP` | 0.05 | макс. шаг вверх (×g) |
| `MARGIN` | 0.05 → 0.02 при streak | надбавка над прогнозом базы |
| **`S_MAX`** | **0.08** | **потолок суммарного сберегательного давления (owner: консервативно)** |
| `G_UP` (breach) | 0.25 | порог «сильного» перебора → rollback |
| `FLOOR` | `max(floor_eur, 0.6·median12)` | пол лимита (не ratchet от median) |
| `STREAK_UNDER` | 2 | мес в рамках → доп. поджатие |
| `STREAK_BASE_UP` | 3 | мес роста базы → разрешить up-move (anti-drift) |
| `CUSUM_K`, `CUSUM_H` | 0.5·σ_MAD, 4·σ_MAD (5 для lumpy) | детектор сдвига на остатках |
| `COV_LO`, `COV_HI`, `G_MIN` | 0.4, 2.0, 0.3 | turbulence-gain `g=clamp(1−(CoV−0.4)/1.6, 0.3, 1)` |
| `LUMPY_QUANTILE` | 0.80 | перцентиль годового envelope |
| `WARMUP` | 6 мес | до этого адаптация выключена (cold-start) |

Полные формулы закона (per-archetype update rule, signal detection, guardrails) — `docs/adaptive-budgets-plan.md` §«Формулы RBAR».

## 6. API contract

Auth: JWT (Bearer) для всех `/v1/web/budgets/*`. Все суммы — EUR (как SPEC-020). Изменения bootstrap — additive.

### `GET /v1/web/budgets/recommendations?period=YYYY-MM` (новый)
Возвращает по каждой активной расходной категории: архетип, текущий лимит, рекомендацию (для recurring/seasonal) ИЛИ конверт (для lumpy), причину. `period` опционально (default = текущий месяц).
```jsonc
{
  "ok": true,
  "period": "2026-06",
  "currency": "EUR",
  "categories": [
    {
      "category_id": "food", "name": "Еда", "emoji": "🍔",
      "archetype": "recurring",
      "current_limit_eur": 520,            // из budgets (SPEC-020), null если нет
      "recommended_limit_eur": 491,
      "delta_pct": -5.6,
      "reason_code": "SAVINGS_STREAK",
      "reason_text": "3 мес в рамках бюджета, база снижается",
      "confidence": "ok",                  // ok | low (E7/E8)
      "baseline_eur": 469,                 // B_next (для прозрачности/графика)
      "floor_eur": 282
    },
    {
      "category_id": "electronics", "name": "Техника", "emoji": "💻",
      "archetype": "lumpy",
      "envelope": {
        "annual_eur": 1130, "accrual_monthly_eur": 94,
        "accrued_eur": 620,                // «накоплено / доступно потратить сейчас»
        "spent_trailing_12m_eur": 510,
        "alert": false                     // true если trailing-12m > annual·1.15
      }
    }
    // fixed → recommended=null, может содержать change_point: {detected:true, from:650, to:720}
  ]
}
```
- Чистая функция `rbar.ts computeRecommendations(history, ratesIndex, settings, period)` — переигрывает историю, считает всё (G7). Переиспользует `toEurAt` (ADR-014) и окно месяца как SPEC-020.

### `GET /v1/web/budgets/archetypes` (новый)
Классификация всех активных расходных категорий + метрики (для UX «как система видит категории», §4.5).
```jsonc
{ "ok": true, "categories": [
  { "category_id": "food", "archetype": "recurring", "archetype_override": null,
    "metrics": { "n_months": 28, "median_eur": 339, "cov_resid": 0.31,
                 "zero_frac": 0.0, "trend_pct_mo": 1.4, "spike": false } }
]}
```

### `PUT /v1/web/budgets/settings/:categoryId` (новый)
Body (Zod `budgetSettingsSchema`): `{ archetype_override?: enum|null, floor_eur?: number|null, adaptive_enabled?: boolean }`. UPSERT в `budget_settings`. 200 `{ ok:true }` | 400.

### `POST /v1/web/budgets/recommendations/:id/decision` (новый)
Body: `{ decision: 'accepted'|'dismissed' }`. Логирует решение в `budget_recommendation_log` (`decided_at`). **При `accepted`** — НЕ дублирует мутацию лимита: клиент отдельно зовёт существующий `PUT /v1/web/budgets/:id` (SPEC-020) с `recommended_limit_eur`; этот endpoint только пишет аудит. (Альтернатива — атомарный accept+apply на сервере; см. OQ4.) 200 `{ ok:true }`.

### `GET /v1/bootstrap` (изменено, additive)
Добавляется lumpy-конверт для read-only lens Mini App (поверх `budgets` из SPEC-020):
```jsonc
"budget_envelopes": [
  { "category_id": "electronics", "accrued_eur": 620, "annual_eur": 1130 }
]
```
Тот же расчётный модуль (`rbar.ts`), auth — существующий initData.

## 7. UI / UX

### Web Admin — расширение `/budgets`
- У каждой строки бюджета (SPEC-020) — **бейдж архетипа** (пилюля: `recurring`/`lumpy`/`fixed`/`seasonal`/`cold-start`, цвет по типу).
- **Карточка рекомендации** (recurring/seasonal, если `recommended != current`): «€520 → **€491** ↓ −5.6%», стрелка вниз=зелёная (экономия)/вверх=амбер (рост базы), `reason_text`, кнопки **«Применить»** / **«Скрыть»**. Прозрачность: тултип/раскрытие с `baseline_eur`, `floor_eur` («есть куда жать до €282»).
- **Lumpy-блок** (конверт): мини-бар «накоплено €620 / годовой €1130», подпись «отчисление €94/мес · доступно €620», алерт-маркер если перерасход годового.
- **Fixed**: change-point баннер (если есть), иначе просто бейдж `fixed` без рекомендации.
- **Секция «Как система видит мои категории»** (сворачиваемая): таблица архетипов + метрики; инлайн-override архетипа (`Select`) и ручного пола (поле EUR); `adaptive_enabled` тумблер. Объясняет и даёт контроль (E2/E7).
- Состояния: `isLoading` скелетон; `isError`→`ErrorState(onRetry)` (Фаза 1.2); cold-start → «собираю данные N/6 мес».
- Инвалидация после accept: `['budgets']`, `['budget-recommendations']`, `['dashboard']`.
- **Advisory-визуал:** рекомендация всегда явно «предложение», не свершившийся факт. Пунктир зарезервирован за forecast (memory `dashed-line-means-forecast`) — baseline-прогноз рисуем пунктиром, рекомендованный лимит — сплошной но в «предложенном» стиле (амбер-рамка).

### Mini App — read-only lens (расширение SPEC-020)
- Для lumpy-категории на плитке: вместо месячного остатка — **«конверт ≈ €620»** (muted). Источник — `bootstrap.budget_envelopes`. Без CRUD (NG7).

## 8. Security

- Все `/v1/web/budgets/*` — за `requireAdminSession` (Bearer JWT, allowlist email). Два auth-канала не смешиваются (правило 4).
- Bootstrap (`budget_envelopes`) — за существующим initData HMAC; суммы — производные от уже доступных трат, новой PII нет.
- Валидация: Zod (shape) + доменные guard'ы (категория существует/active/`type='expense'`; `archetype_override` из enum; `floor_eur ≥ 0`; `period` формат `YYYY-MM`). Ошибки → 400 без stack-leak.
- Без новых secrets. SQL параметризован. В логи не пишем суммы/историю.
- Авто-применение отсутствует (G5) → нет риска «машина молча увела лимит».

## 9. Acceptance criteria

> Полный список разбит по фазам в `docs/adaptive-budgets-plan.md`. Здесь — фичевые AC (что значит «механизм работает правильно»).

- [ ] **AC1** (классификатор): `classifyArchetype(history)` корректно маркирует на наших данных: Жильё→fixed, Еда/Продукты/Досуг→recurring, Коммуналка→seasonal(или recurring при <24мес), Техника/Одежда/Дом→lumpy, Подписки/Чаевые→intermittent, `<6мес`→cold-start. Метрики на **детрендированном** ряде (сырой CoV не используется — он врёт при тренде).
- [ ] **AC2** (база): damped-Holt `B_next` трекает растущий ряд без лага-ratchet; на симуляции «плоский EWMA» проваливается, Holt — нет (регресс-тест на зафиксированном векторе).
- [ ] **AC3** (винзоризация): месяц-всплеск (€606 Еда / €4503 Техника) не попадает в базу (clip ±3·σ_MAD) → рекомендация recurring не обваливается.
- [ ] **AC4** (экономия → вниз): 3 мес снижающихся трат → рекомендованный лимит снижается шагами ≤2.5%/мес, в пределах `S_MAX=8%` суммарно, не ниже floor.
- [ ] **AC5** (асимметрия): разовый перебор без подтверждённого роста базы 3 мес → **HOLD** (лимит не меняется), `reason=HOLD_AFTER_BREACH`. Перебор >25% → откат к `L_comfort`, `reason=ROLLBACK`.
- [ ] **AC6** (anti-drift, ГЛАВНОЕ): рост лимита разрешён ТОЛЬКО при `streak_baseline_up≥3`. Регресс-тест: на шумном ряде без тренда лимит **не дрейфует вверх** (Δ итог за 24 мес ≈ 0 ± малое), в отличие от наивной асимметрии (+23–35%).
- [ ] **AC7** (turbulence): высокий CoV → меньший шаг (g масштабирует). Досуг (CoV0.66, g≈0.84) двигается медленнее Еды (CoV0.41, g≈1.0) на тех же сигналах.
- [ ] **AC8** (lumpy конверт): для категории с `zero_frac≥0.4` месячный лимит НЕ считается (нет div-by-zero); считается годовой envelope (Q0.80 скользящих 12-мес сумм), месячное отчисление, накоплено, алерт при trailing-12m > annual·1.15.
- [ ] **AC9** (детерминизм/идемпотентность, G6): один и тот же `history` + `period` → одинаковый результат; до-ввод траты задним числом меняет рекомендацию консистентно (полный реплей, без накопленного стейта).
- [ ] **AC10** (advisory, G5): эндпоинты НЕ меняют `budgets.limit_eur`. Применение — только явный `PUT` (SPEC-020). `decision`-endpoint лишь логирует.
- [ ] **AC11** (override): `PUT settings` сохраняет `archetype_override`/`floor_eur`/`adaptive_enabled`; они учитываются в следующем расчёте; `adaptive_enabled=0` → рекомендаций нет (только ручной лимит SPEC-020).
- [ ] **AC12** (единый модуль, G7): `rbar.ts` переиспользован `/recommendations`, `/archetypes` и bootstrap'ом (нет копий формул).
- [ ] **AC13** (бэктест, G8): тест прогоняет RBAR по всем recurring-категориям на 29 мес истории, выводит траекторию `L_t` и метрику дрейфа; проверяет отсутствие апдрейфа и что floor ни разу не пробит вниз нереалистично.
- [ ] **AC14** (UI Admin): бейджи архетипов, карточки рекомендаций (применить/скрыть), lumpy-конверт, секция «как система видит категории» с override; loading/error/cold-start; toast + инвалидация. Скриншоты light/dark (memory `frontend-test-locally-before-deploy`).
- [ ] **AC15** (UI Mini App): read-only lens конверта на плитке lumpy-категории; без CRUD; light/dark.
- [ ] **AC16** (vitest): покрыт классификатор, damped-Holt, winsorize, CUSUM, slew, asymmetry+anti-drift, lumpy envelope, cold-start, turbulence-gain, реплей-детерминизм, бэктест-вектор.
- [ ] **AC17** (docs): миграция 0012 + `schema.sql`; `gitleaks dir` clean; ADR-016 (закон RBAR) черновик; roadmap 2.6 обновлён; CLAUDE.md §11 уточнён (Mini App lens конверта).

## 11. Risks & open questions

- **R1** (anti-drift консервативен): рост лимита с лагом 3–4 мес + damped занижает ~18%. В advisory это лишь чуть заниженная рекомендация, не блокировка. Митигация: rollback при сильном переборе; tunable `STREAK_BASE_UP`.
- **R2** (over-tuning, неустраним): 8–12 констант на `N=12–29`. Поэтому бэктест (AC13) — обязательный gate, а не косметика; параметры — глобальная политика, не fit (NG4).
- **R3** (граница lumpy↔recurring): Здоровье/Подарки могут мигрировать архетип при до-вводе. Митигация: гистерезис 15% + ручной `archetype_override`.
- **R4** (seasonal на грани): Коммуналка ~24–29 мес, на пределе идентифицируемости сезонности. До 24 мес — recurring с широким коридором; реальная сезонная защита через год+.
- **R5** (perf): полный реплей истории на каждый запрос. Single-user, 29мес×25кат — копейки; запрос трат — точечный (как SPEC-020 G6). Кэш не нужен.
- **R6** (двойной счёт): G4/линза устраняет; ни конверт, ни профицит не создают тратимый баланс. Документируем в doc-comment `rbar.ts`.
- **OQ1** (resolved): применение — **advisory-first** (owner 2026-05-31).
- **OQ2** (resolved): конверт/профицит — **линза-подсказка**, без отдельного баланса (owner 2026-05-31).
- **OQ3** (resolved): агрессивность — **консервативная, `S_MAX=8%`** (owner 2026-05-31).
- **OQ4**: `accept` рекомендации — атомарный server-side (decision+apply одним запросом) или клиент делает 2 вызова (decision-log + PUT)? Спека закладывает 2 вызова (проще, переиспользует SPEC-020); атомарность — если на практике будет рассинхрон.
- **OQ5**: «в рамках бюджета» для streak — по факту закрытого месяца (детерминированно) или прогноз run-rate? Спека: **по закрытому месяцу** (детерминизм важнее раннего сигнала).
- **OQ6**: гранулярность — лимит на категорию верхнего уровня (сейчас) или подкатегории (если появится иерархия)? Вне scope, отметка на будущее.
- **OQ7**: профицит-линза («сэкономил €X/год») — показывать в этой итерации или отдельной? Спека: **отложить** в Фазу 4 плана (фокус v1 — рекомендации + конверт).
