# График инвестиций — выбор периода

> Фидбэк owner (2026-06-09): «график в разделе инвестиций не настраивается и показывает просто за
> 12 месяцев, хотя ожидается такой же функционал как на других дашбордах с разбивкой по времени».
> Owner-решение по scope: **period selector к существующим спарклайнам** (не отдельный крупный график).

## 1. Context & Problem

Спарклайн стоимости в карточке инвест-позиции (`InvestmentsPage` → `value_series`) жёстко показывает
12 месяцев — нет выбора периода. На дашборде и в списках есть привычный контрол периода (`PeriodPresets`
на `/dashboard`: 12м/6м/год/всё/период → серверный диапазон `from/to`). Инвестиции из этого паттерна
выпадают.

## 2. Goals

- **G1**: на странице «Инвестиции» — контрол периода **12м / 6м / Год / Всё / Период** (тот же набор и
  вид, что `PeriodPresets` дашборда).
- **G2**: спарклайн каждой позиции (`value_series`) строится за **выбранное окно**, а не за фикс. 12 мес.
- **G3**: переиспользуем серверный паттерн дашборда — `from/to` query → worker генерит месячный ряд в
  диапазоне (без копий логики конверсии/баланса).

## 3. Non-Goals

- **NG1**: отдельный крупный line-chart суммарной стоимости портфеля (как net worth на дашборде) — НЕ
  в этой итерации (owner выбрал минимальный вариант; можно отдельной задачей по сигналу).
- **NG2**: разбивка спарклайна по форме/валюте (как nwMode дашборда) — не нужна (одна крипто-позиция).
- **NG3**: смена гранулярности (недельная/дневная) — остаёмся помесячно, как дашборд.
- **NG4**: изменение KPI «Сейчас» (стоимость/cost basis/P&L/доход) — они mark-to-market, период их не трогает.

## 4. User journeys

### Happy path
1. Stepan открывает «Инвестиции» → дефолт период **12 мес** (как сейчас).
2. Жмёт «6 мес» → спарклайн позиции перерисовывается за последние 6 мес (6 месячных точек).
3. «Всё» → ряд с первого месяца с данными позиции; «Период» → два date-инпута (custom from/to).
4. Прогноз APR-пунктиром (для застейканных) остаётся — добавляется от последней фактической точки.

### Edge cases
- **E1**: период короче истории позиции (6м, а позиция новее) → ряд от первого доступного месяца (≤ выбранных точек), без «пустых» нулей слева до появления позиции — как сейчас (значение 0 до первой операции).
- **E2**: custom `from > to` или пустые → фронт не шлёт некорректный диапазон (валидация инпутов как на дашборде: `max=to`, `min=from`), worker валидирует ISO и игнорирует мусор (fallback на дефолт-окно).
- **E3**: «Всё» при длинной истории (с 2024) → ряд помесячно, без переполнения (cap итераций).
- **E4**: custom-период целиком в прошлом (`to < today`) → ряд по [from..to]; forecast-пунктир для staked рисуется от последней точки (концептуально для прошлого периода не показателен — допустимо, как и спарклайны дашборда).

## 5. Data model

Изменений в D1 нет. `value_series` — by-read линза (`investments.ts`), параметризуется окном.

## 6. API contract

### `GET /v1/web/investments` (расширение)
- Auth: JWT (Bearer) — без изменений.
- Query (добавлено): `from`, `to` (YYYY-MM-DD, опц.). Семантика как у `/v1/web/dashboard`: окно
  для `value_series`. Без них — дефолт 12 мес (обратная совместимость). `today` — как раньше.
- Response 200: та же форма; `positions[].value_series` теперь покрывает [from..min(to,today)] помесячно.

Внутри: `getInvestments(env, { today, from?, to? })` — `value_series` итерируется от `monthKey(from ?? today−11м)`
до `monthKey(min(to, today))` помесячно (конец месяца, cap 240 итераций). KPI/cost basis/P&L — без изменений.

## 7. UI / UX

`InvestmentsPage`: контрол `PeriodPresets` в шапке (справа, рядом с refresh) — как в превью owner'а.
Состояние периода на уровне страницы (общий для всех карточек). Спарклайн в `PositionCard` рисует
`value_series` пришедшего окна. `PeriodPresets` выносится в общий компонент `components/PeriodPresets.tsx`
и переиспользуется дашбордом (рефактор без смены поведения).

## 8. Security

- Только чтение; `from/to` — валидация ISO-формата, в SQL не уходят как сырые (series считается in-memory
  из уже загруженных tx/snapshots; конверсия через canonical `RatesIndex`).
- Auth — существующий admin JWT. Новых поверхностей нет.

## 9. Acceptance criteria

- [ ] AC1: на «Инвестициях» есть контрол 12м/6м/Год/Всё/Период (тот же компонент, что на дашборде).
- [ ] AC2: смена периода перерисовывает спарклайн позиции за выбранное окно (кол-во точек = месяцев окна).
- [ ] AC3: `GET /v1/web/investments?from=&to=` возвращает `value_series` в диапазоне; без параметров —
  дефолт 12 мес (обратная совместимость).
- [ ] AC4: `getInvestments` с разными окнами даёт корректную длину/границы ряда (unit-тест).
- [ ] AC5: KPI «Сейчас» (стоимость/cost basis/P&L/доход стейкинга) не зависят от периода (не меняются).
- [ ] AC6: `PeriodPresets` вынесен в общий компонент; дашборд работает как прежде (визуал-регресс зелёный).
- [ ] AC7: forecast-пунктир (staked) остаётся и корректно добавляется от последней точки выбранного окна.

## 10. Test plan

- **Worker (vitest)**: `investments.test.ts` — `getInvestments({from,to})`: длина ряда для 6м/12м/custom,
  границы (первый/последний месяц), cap на «всё», дефолт без параметров.
- **Regression**: весь `vitest run` + typecheck (worker + admin).
- **Admin SPA**: Playwright — investments при разных периодах (12м/6м/custom) light/dark; дашборд не съехал.

## 11. Risks & open questions

- **R1**: рефактор `PeriodPresets` из `DashboardPage` может задеть рабочий дашборд. Митигация: вынос
  без смены поведения (тот же тип/диапазоны), визуал-регресс дашборда в харнесе.
- **OQ1**: нужен ли в будущем крупный график портфеля (NG1)? — по сигналу owner'а.
