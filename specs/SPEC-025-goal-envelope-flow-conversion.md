# Вклад в цель — поток (date-aware), а не псевдо-запас; честный free

## 1. Context & Problem

Доход можно привязать к цели (`incomes.goal_id`), цель копит в своём `target_currency`. Сейчас баланс цели считается **mark-to-market по курсу исходной валюты вклада НА СЕГОДНЯ** (`goals.ts:listGoals` → `convertAt(amount, currency, target, today)`). Это работает, пока деньги лежат в той же валюте, в которой пришёл доход. Но как только целевые деньги обменены (RUB → USDT → EUR), они физически уходят в другую валюту, а цель **продолжает переоценивать их по курсу рубля, которого уже нет на руках**.

Два следствия:
1. **Прогресс цели в EUR болтается по курсу валюты, которой у вас уже нет.** Вы ушли в USDT, чтобы зафиксировать стоимость, — система этого «не заметила».
2. **В `free = net − targeted` протекает курсовой рассинхрон** между «валютой исходного дохода» (по ней считается `targeted`) и «валютой, где деньги реально лежат» (по ней считается `net` через `effective_balance` ведра). После каждого обмена появляются фантомные «свободные» или «недостающие» евро.

По сути текущий MtM — **псевдо-запас**: он применяет курс исходной валюты дохода, а не валюты текущего нахождения денег (которую мы сознательно не трассируем — см. NG1). Настоящий запас невозможен без трассировки. Правильная классификация: **вклад в цель — это поток** (событие на дату), как расход и доход (ADR-014). Это пересматривает решение SPEC-016 R3 (MtM goal balance, 2026-05-27): тогда проблема обменов не рассматривалась.

## 2. Goals

- **G1**: Вклад в цель (`incomes.goal_id` и `goal_contributions`) конвертируется в `target_currency` по курсу **на дату вклада** (поток), фиксируется и не зависит от будущих курсов и обменов.
- **G2**: Обмен валюты (`transactions`) **не меняет** баланс ни одной цели (закрепляется regression-тестом; структурно уже так — цель не читает `transactions`).
- **G3**: `free = net − targeted` — честный индикатор обеспеченности целей капиталом; может быть **отрицательным** (= цели недообеспечены), нигде не клампится на worker, в Admin показывается danger-сигналом.
- **G4**: Прогресс цели в `target_currency` стабилен во времени при неизменных вкладах (не «дрожит» с дневным курсом).
- **G5**: **Без миграции D1** — меняется только формула расчёта on-read.

## 3. Non-Goals

- **NG1**: НЕ трассируем целевые деньги через обмены (lot/lineage tracking). Это явно отвергнутый вариант (owner-решение 2026-06-01: «доделать конверт», не «трассировка»). Спящая `transactions.goal_id` остаётся спящей.
- **NG2**: НЕ вводим пообъектную обеспеченность («ипотека обеспечена, отпуск — нет») — только суммарный `free`. Конверт даёт только агрегат.
- **NG3**: НЕ меняем конверсию `target_currency → EUR` для `balance_eur`/`targeted_eur` (она остаётся `today`) — это корректный запас для EUR-сводки, обоснование в §5.
- **NG4**: НЕ трогаем `net_worth_series` и free-спарклайн (`DashboardPage:216`, кламп `Math.max(0, …)` — аппроксимация формы тренда, historical goal balance не реконструируем).
- **NG5**: НЕ добавляем новый UI-блок «доложить до обеспечения» — только существующий danger-сигнал (опционально-текст вынесен в OQ1).

## 4. User journeys

### Happy path
1. Записал доход 100 000 RUB в цель «ипотека» (`target_currency = EUR`) с датой 2024-01-01.
2. Система фиксирует вклад как EUR-эквивалент **по курсу на 2024-01-01** — дальше он не меняется.
3. Обменял RUB → USDT, позже USDT → EUR. Баланс цели «ипотека» не меняется; `free` не дёргается от обменов (с точностью до спреда — реальной цены обмена).
4. Открыл дашборд: прогресс ипотеки в EUR стабилен и отражает зафиксированные вклады. Если суммарный капитал просел ниже обещанного целям — «Свободно» отрицательное (сигнал «доложить / пересмотреть цели»).

### Edge cases
- **E1**: На дату вклада нет точного курса → `rateAt` берёт ближайший `≤ дате` (fallback назад, не 0). Нет вообще ни одного `≤` → позиция уходит в `balance_missing_rates` (как сейчас, но по дате вклада).
- **E2**: Валюта вклада == `target_currency` → конверсия = identity (×1), дата не важна.
- **E3**: Вклад с очень старой датой (раньше первого курса в `rates`) → ближайший доступный курс через fallback; край маловероятен (rates backfilled).
- **E4**: Legacy-цель без `target_currency` → EUR-нейтраль (как сейчас), но конверсия по дате вклада.
- **E5**: Backdated income (экономическая дата в прошлом, `created_at` сегодня) → курс берётся на **экономическую дату** — консистентно с потоком расход/доход (ADR-014, SPEC-024).

## 5. Data model

**Схема и миграции — без изменений.** Меняется только формула расчёта on-read:

- `goals.ts:listGoals` — в SELECT из `incomes` и `goal_contributions` добавить `date`; в агрегации использовать `convertAt(amount, currency_code, target, row.date)` вместо `today`.
- `goals.ts:getGoalDetail` — timeline уже содержит `date`; заменить `today` → `r.date` (стр. 220, 227); `delta_in_target` каждой строки фиксируется по её дате.
- `balance_eur` / `targeted_eur` (`goals.ts:182-183`, `dashboard.ts:183`, `index.ts:363`) — **остаётся `today`**.

**Почему `balance → EUR` остаётся `today` (запас), а вклад → `target_currency` становится потоком (по дате):**

Это два разных шага с разной природой:
- Шаг 1 (вклад → `target_currency`) — **поток**: «сколько единиц целевой валюты я отложил тогда». Фиксируется по дате вклада. Это и есть прогресс цели.
- Шаг 2 (накопленное в `target_currency` → EUR для общей сводки) — **запас**: «сколько целевое стоит в EUR сейчас», по `today`. Нужен лишь чтобы свести разные цели и net worth в одном EUR-табло.

Инвариант `free = net − targeted` при этом честный: `net` следует за валютой, где деньги **реально лежат** (`effective_balance` ведра учитывает обмены), `targeted` зафиксирован в целевой валюте. Если цель и деньги в одной валюте — курс этой валюты к EUR влияет на `net` и `targeted` одинаково, `free` стабилен. Если разошлись — расхождение и есть сигнал обеспеченности.

**Семантика:** вклад в цель переклассифицируется из «запас» в **«поток»** (date-aware) — уточнение ADR-014. Фиксируется в **ADR-020** (номер ADR-018 из плана уже занят инвестициями).

## 6. API contract

Shape ответов **не меняется**. Меняются только **значения** денежных полей в существующих ответах:
- `GET /v1/web/goals` — `balance`, `balance_eur`, `target_amount_eur`.
- `GET /v1/web/goals/:id` — `balance`, `contributions[].delta_in_target`.
- `GET /v1/web/accounts` (summary) — `targeted_eur`, `free_eur`.
- `GET /v1/web/dashboard` — `targeted_eur`, `free_net_worth_eur`, `prev_free_net_worth_eur`.

Auth: JWT (Bearer) на всех — без изменений. **Важно:** после деплоя значения сдвинутся (пересчёт вкладов по историческим курсам вместо сегодняшнего) — см. R1.

## 11. Risks & open questions

- **R1**: После деплоя баланс **всех существующих целей** пересчитается (вклады в не-целевых валютах зафиксируются по историческим курсам вместо сегодняшнего) → цифры заметно сдвинутся, особенно по рублёвым вкладам (курс сильно двигался 2024–2026). Это **ожидаемо** — это и есть цель фичи. Проверить до/после на проде; без миграции, откат = revert формулы.
- **R2**: Пересмотр недавнего явного решения owner'а (SPEC-016 R3, MtM, 2026-05-27). Зафиксировать в ADR-020 с обоснованием (проблема обменов тогда не рассматривалась).
- **R3**: Доступность исторических курсов на старые даты вкладов — покрыто `rateAt` fallback назад; край (вклад раньше самого раннего курса) маловероятен.
- **OQ1**: Нужен ли явный текстовый индикатор «цели недообеспечены на X €» сверх danger-цвета? Предлагаю post-MVP, не блокер этой итерации.
