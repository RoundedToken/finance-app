# SPEC-050 · Drift-журнал (кандидат, BP-01 аудита 2026-07)

> **Draft / SPEC-кандидат.** Единственное структурное отставание системы от
> Beancount/YNAB-класса по вердикту аудита (отчёт 10, BP-01, P1-стратегический).
> Реализация — после owner-решения по Open Questions.

## 1. Context & Problem

Manual snapshot становится новым baseline (SPEC-011) и **молча поглощает** невязку
между расчётным и фактическим балансом. После записи снапшота нельзя ответить
«сколько трат я не записал в июне». Балансы верны, но burn / норма сбережений /
бюджеты / RBAR-рекомендации систематически приукрашены на величину недозаписи —
и это невозможно обнаружить постфактум. У зрелых систем невязка — first-class
запись: Beancount `pad`, Firefly reconciliation-транзакция, YNAB Balance Adjustment.

## 2. Design (эскиз из аудита)

1. Миграция: `snapshots.drift REAL NULL`; при POST/PUT снапшота worker пишет
   `drift = amount − effective_balance(bucket, date)` на момент записи.
2. Форма снапшота в Admin показывает drift ДО сохранения («расчётно 1 234.56,
   вводишь 1 200.00, разница −34.56 — ок?»).
3. Лента дрейфов на странице ведра; на дашборде — месячная строка
   «неучтённые потоки ≈ X €» (Σ drift, конверсия потоком по дате снапшота).
4. Coach-нудж (SPEC-040) при |drift| выше порога.
5. Максимум-производная: месячный cross-check `ΔNW ≈ income − expenses + fx + drift`
   как вторая, balance-based оценка нормы сбережений (связка с BP-04).

## 3. Open Questions (owner)

- OQ1: drift показывать только в Admin или и в Mini App при вводе снапшота? (Mini App снапшоты не вводит — только Admin; но нудж-канал общий.)
- OQ2: порог coach-нуджа (фикс € или % от оборота ведра?).
- OQ3: включать ли п.5 (cross-check нормы) в первую итерацию или отдельно с BP-04.

## 4. Оценка

M (миграция S + worker S + Admin UI M). Backfill исторических drift невозможен
(эффективный баланс на момент прошлых снапшотов уже перезаписан ими самими) —
журнал начинается с деплоя.
