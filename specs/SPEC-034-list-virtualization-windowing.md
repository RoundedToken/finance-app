# Виртуализация длинных списков (windowing)

## 1. Context & Problem

Два списка грузятся целиком и **рендерятся в DOM полностью**: Admin `/expenses` (`limit=20000`, TanStack Table, `table.getRowModel().rows.map(...)`) и Mini App `HistoryScreen` (`limit=5000`, группировка по дням, `days.map → rows.map`). Сейчас данных ~1.8k трат и они «рендерятся мгновенно», но число DOM-узлов растёт линейно с историей (особенно тяжело на iPhone: каждая строка — `SwipeRow` со своими touch-хендлерами). Это профилактический долг под рост данных, зафиксированный в `docs/roadmap.md` («Виртуализация длинных списков») и `post-mvp-roadmap.md` (3.4).

Исходная формулировка фичи была «keyset-пагинация + react-virtual». Discovery (4 параллельных Explore-агента, 2026-06-26) показал: **все фильтры/сортировка/поиск/KPI в Admin — клиентские по полному in-memory массиву**, поэтому серверная пагинация без переноса фильтров на сервер бессмысленна (клиенту всё равно нужны все строки, чтобы фильтровать), а перенос фильтров+KPI на сервер — в разы больший объём, трогающий деньги-контракты, и преждевременный для 1.8k строк. Решение владельца (2026-06-26): **scope = только windowing (рендер)**; keyset-пагинация — отдельным SPEC по реальному триггеру (рост данных + перенос фильтров на сервер).

## 2. Goals

- G1: На Admin `/expenses` в DOM присутствуют только видимые строки + overscan-буфер, а не все N (проверяемо: `document.querySelectorAll` по строкам таблицы ≪ числа отфильтрованных записей при N≫viewport).
- G2: На Mini App `HistoryScreen` в DOM присутствуют только видимые блоки-дни + overscan, а не все дни истории.
- G3: **Никаких изменений деньги-математики и контрактов API.** Worker, миграции, формат ответов list-эндпоинтов — не трогаем. Полный load сохраняется.
- G4: Корректность сумм сохранена: Admin KPI (`totalEur`/`byCurrency`/«Записей») и Mini App `DayTotal` считаются по полному набору (как сейчас), а не по видимому окну — SPEC-033 не нарушается.
- G5: Скролл плавный, при прокрутке нет «провалов»/пустот (overscan + динамическое измерение высоты строк).
- G6: Все текущие состояния и поведение сохранены: loading / error (`ErrorState` + retry) / empty, сортировка по клику на заголовок, фильтры/поиск/период, swipe-delete и tap-to-edit (Mini App), hover-подсветка строк (Admin).

## 3. Non-Goals

- NG1: Серверная keyset/offset-пагинация и любые изменения Worker/D1/миграций. Отдельный будущий SPEC.
- NG2: Перенос фильтров/сортировки/поиска/KPI на сервер. Остаются клиентскими.
- NG3: Виртуализация прочих списков (снапшоты/обмены/взносы — десятки строк, доходы — сотни). Не входит: выигрыша нет, золочение.
- NG4: `useInfiniteQuery` / infinite scroll / подгрузка по мере прокрутки. Грузим всё одним запросом, как сейчас; виртуализируем только рендер.
- NG5: Browser native find-in-page (Ctrl+F) по невидимым строкам перестаёт находить скрытые окном строки — осознанный trade-off виртуализации; в Admin уже есть поле поиска, в Mini App поиска нет.
- NG6: Mini App scope не расширяется (правило 11): это perf-рефактор существующего экрана истории, не новая функциональность.

## 4. User journeys

### Happy path — Admin Расходы
1. Открываю `/expenses`. Таблица показывает строки в bounded scroll-области с прилипшим (`sticky`) заголовком.
2. Скроллю вниз — строки догружаются в DOM по мере прокрутки без задержки, заголовок остаётся на месте, KPI-карточки сверху неизменны.
3. Кликаю заголовок «Сумма» — сортировка применяется ко всему набору, окно перерисовывается с новой вершиной.
4. Ввожу фильтр по категории/период/поиск — список и KPI пересчитываются по полному набору, окно показывает отфильтрованные строки.

### Happy path — Mini App История
1. Открываю «История». Вижу блоки по дням (заголовок дня + `DayTotal` + строки), как раньше.
2. Скроллю — старые дни появляются в DOM по мере прокрутки, ранее проскролленные — выгружаются. Визуально идентично прежнему (rounded-карточки дня, разделители).
3. Свайпаю строку влево → удалить; тап → редактирование. Работает как раньше.

### Edge cases
- E1: 0 записей → empty-state (`Пока нет трат` / `Нет записей под текущие фильтры`), виртуализатор не активен.
- E2: Загрузка → loading-state; ошибка 5xx → `ErrorState`+retry (Admin) / текущий loading (Mini App). Виртуализатор не активен.
- E3: Один день с большим числом трат (bulk-импорт) → блок дня измеряется `measureElement` и корректно занимает высоту; рендерятся все строки этого дня, когда он в окне (приемлемо — ограничено одним днём).
- E4: Строка с длинным `note`, переносящимся на 2 строки → динамическая высота через `measureElement`, без наложений.
- E5: Сортировка/фильтр меняют число строк → `count` виртуализатора обновляется, спейсеры пересчитываются.

## 5. Data model

Изменений в D1 нет. Миграции не добавляются. (Подтверждение для review: NG1.)

## 6. API contract

Изменений нет. Списки грузятся существующими эндпоинтами в текущем формате:
- `GET /v1/web/expenses?limit=20000` → `{ expenses: Expense[] }` (Admin).
- `GET /v1/expenses?limit=5000` → `{ expenses: Expense[] }` (Mini App).

Никаких новых query-параметров (`cursor`/`after`), никакого `nextCursor` в ответе.

## 7. UI / UX

Новая зависимость: `@tanstack/react-virtual@^3` (в `cloud/admin` и `cloud/miniapp`). Совместима с React 19 (v3.14.4 на npm).

### Admin `/expenses`
- Контейнер таблицы (`card`) получает bounded высоту с вертикальным скроллом: `max-h-[calc(100vh-360px)]` (минимум ~`min-h-[300px]`), `overflow-auto`. Этот контейнер — `getScrollElement` для `useVirtualizer`.
- `<thead>` → `sticky top-0 z-10` (прилипает при вертикальном скролле внутри контейнера).
- Тело: padding-row техника (сохраняет распределение ширин колонок реального `<table>`): спейсер-`<tr>` сверху (высота = `virtualItems[0].start`), затем видимые `<tr>` с `data-index` + `ref={virtualizer.measureElement}`, затем спейсер-`<tr>` снизу. `estimateSize ≈ 44`, `overscan 12`.
- loading/error/empty рендерятся как сейчас (один `<tr colSpan>`), без виртуализатора.
- Горизонтальный скролл колонок (`overflow-x-auto`) сохраняется внутри того же контейнера.

```
┌ card ───────────────────────────────────┐
│ ┌ thead (sticky) ──────────────────────┐ │
│ │ Дата │ Категория │ … │ Сумма │ …     │ │
│ ├──────────────────────────────────────┤ │  ← bounded height, overflow-auto
│ │ [spacer top]                         │ │
│ │ row k   (только видимые + overscan)  │ │
│ │ row k+1 …                            │ │
│ │ [spacer bottom]                      │ │
│ └──────────────────────────────────────┘ │
└──────────────────────────────────────────┘
```

### Mini App `HistoryScreen`
- **Day-level виртуализация**: virtual item = блок одного дня (`<div>`: заголовок дня + `DayTotal` + контейнер строк дня). Сохраняет существующий визуал (rounded-карточка, `divide-y`) один-в-один.
- Экран → `flex flex-col h-screen`; шапка («← История») — первый flex-child; список — `flex-1 overflow-y-auto` (это `getScrollElement`).
- `useVirtualizer({ count: days.length, estimateSize: () => 120, overscan: 4, measureElement })`. Каждый день абсолютно позиционируется (`position:absolute; top:0; left:0; width:100%; transform: translateY(start)`), внешний контейнер списка — `position:relative; height: getTotalSize()`.
- `DayTotal` для каждого дня считается из полного списка трат дня (в памяти) при построении `byDay` — окно влияет только на отрисовку, не на сумму (G4, SPEC-033).
- loading/empty — как сейчас.

## 8. Security

- Auth не меняется (Admin JWT, Mini App initData) — эндпоинты те же.
- Input validation не затрагивается (нет новых входов).
- PII/финданные: тот же набор, в логи/телеметрию ничего нового не уходит.
- Виртуализация — чистый клиентский рендер; новых поверхностей атаки нет.

## 9. Acceptance criteria

- [x] AC1: Admin `/expenses` при N строк ≫ высоты окна — в DOM число строк-`<tr>` (исключая спейсеры/заголовок) близко к (видимые + overscan), а не N.
- [x] AC2: Admin — сортировка по клику на «Дата»/«Сумма»/«EUR-эквив.» работает по всему набору; окно перерисовывается корректно.
- [x] AC3: Admin — фильтр по категории/валюте/периоду и поиск пересчитывают список и KPI (`totalEur`, `byCurrency`, «Записей») по полному отфильтрованному набору (значения совпадают с до-виртуализационным поведением).
- [x] AC4: Admin — `<thead>` остаётся видимым (sticky) при вертикальном скролле тела; ширины колонок не «прыгают».
- [x] AC5: Admin — loading/error(`ErrorState`+retry)/empty отображаются как раньше; hover-подсветка строки сохранена.
- [x] AC6: Mini App — История рендерит только видимые блоки-дни (+overscan) в DOM; при скролле новые дни появляются без пустот.
- [x] AC7: Mini App — `DayTotal` каждого дня = сумма по **всем** тратам дня (SPEC-033 не нарушен), независимо от того, какие дни сейчас в окне.
- [x] AC8: Mini App — swipe-delete и tap-to-edit работают на виртуализированных строках; шапка «История» остаётся сверху.
- [x] AC9: Worker/D1/миграции/контракты API без изменений (diff не трогает `cloud/worker`, `migrations`).
- [x] AC10: `npm run build`/typecheck зелёные в `cloud/admin` и `cloud/miniapp`; существующие vitest (worker) не затронуты и зелёные.

- **Mini App**: Playwright (`local/scripts/test_miniapp_react.py`, mock `/v1/expenses` многодневным набором) — light/dark: рендер истории, скролл, DayTotal-сходимость, swipe/tap. iOS-харнес (`test_miniapp_ios.py`) опционально для swipe.
- **Worker**: не затронут; smoke `npm test` (vitest) — без новых тестов, подтвердить зелёный.
- **Regression**: Admin `/incomes`/`/transactions`/`/snapshots` (не виртуализированы) — без изменений; Mini App `MainScreen.RecentDays` (тоже группировка по дням, не трогаем) — без изменений; SPEC-033 (`RecentDays` итоги) — без регресса.

## 11. Risks & open questions

- R1: **Динамическая высота строк** (перенос `note`, разные дни). Митигировано `measureElement` + разумный `estimateSize`. Без `measureElement` возможны «дёргания» при скролле.
- R2: **TanStack Table + sticky `thead` + padding-row.** Absolute-позиционирование `<tr>` ломает ширины колонок → используем padding-row технику (нормальный табличный поток), не translate на `<tr>`.
- R3: **Mini App window vs container scroll.** Выбран container-scroll (`flex-1 overflow-y-auto`) — проще, без `scrollMargin`-математики `useWindowVirtualizer`; History без инпутов, viewport стабилен.
- R4: **SwipeRow под виртуализацией.** Day-level рендер сохраняет строки в обычном потоке внутри блока дня (не абсолют на уровне строки) → swipe-трансформации не конфликтуют. Проверить в тесте.
- R5: **Browser find-in-page** не находит скрытые окном строки (NG5). Осознанно.
- OQ: нет — scope зафиксирован ответами владельца (windowing-only; Расходы + История).

## 12. Out of scope для review

- Не ругать отсутствие серверной пагинации/курсора (NG1) и клиентские фильтры/KPI (NG2) — сознательно отложено.
- Не ругать невиртуализированные короткие списки (NG3).
- Не требовать `useInfiniteQuery` (NG4).
