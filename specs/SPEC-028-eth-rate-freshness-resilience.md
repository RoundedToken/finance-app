# ETH-курсы — устойчивость фетча и свежесть по времени фетча

> Фидбэк с живого прода (owner, 2026-06-09): «за последние два дня в БД нет рейтов ETH — что-то
> неправильно работает; надо фетчить курс хотя бы 4 раза за сутки; и чтобы в приложении брался
> актуальный курс не с даты, а по времени фетча». Owner-решение по семантике (2026-06-09):
> **внутридневная история** курса (а не «один последний за день»).

## 1. Context & Problem

Крипто-курс ETH фетчится один раз в сутки (cron `0 6 * * *`) только с **Binance** `ETHEUR`, в
отдельном try/catch от фиата. Диагностика прода 2026-06-09:

- Фиат (USD/RUB/RSD/TRY/USDT) записан сегодня `2026-06-09 06:00:16` → **cron жив и отрабатывает**.
- ETH последний раз записан `date=2026-06-07, fetched=2026-06-07 06:00:12`. За 06-08 и 06-09 ETH **нет**.
- Код не менялся с 06-06 (деплой SPEC-027) и 06-07 ETH записался успешно. Binance `ETHEUR` жив с
  внешнего IP (отдаёт ~1446 EUR). Значит сломалось **внешнее**: Binance начал отвергать запросы именно
  с Cloudflare-IP (материализовавшийся риск R1 из SPEC-026 — гео-блок/IP-ban). Падение Binance молча
  гасится try/catch → ETH остаётся «вчерашним», и так уже два дня.

Три проблемы: (1) **single point of failure** — один провайдер, упал → курса нет; (2) **редкий фетч** —
1×/сутки мало для волатильного актива; (3) **семантика чтения** — стоимость портфеля берёт курс «на
календарный день», а не самый свежий по факту фетча; при отставшем наполнении показывается устаревшая цена.

## 2. Goals

- **G1** (частота): cron фетчит курсы **4 раза в сутки** (`0 */6 * * *` → 00/06/12/18 UTC).
- **G2** (устойчивость): крипто-курс берётся по **цепочке провайдеров** Binance → Coinbase → CoinGecko;
  первый успешный побеждает. Падение одного не оставляет систему без курса (убираем SPOF, mitigation R1).
- **G3** (свежесть «по времени фетча»): хранится **внутридневная история** курса (`rate_ticks`, по одной
  строке на фетч с `fetched_at`); стоимость портфеля «сейчас» (mark-to-market) берёт **самый свежий тик**,
  а не курс на календарную дату. Историческая конвертация (траты/доходы/cost basis по дате операции) —
  без изменений (поток по дате остаётся дневным).
- **G4** (видимость): `refresh-rates` возвращает, **какой провайдер** сработал и текст ошибки если все
  упали; на странице инвестиций видно **время последнего фетча** курса.

## 3. Non-Goals

- **NG1**: внутридневная история для **фиата** — не нужна (Google Sheets обновляет фиат ~раз в день;
  фиат остаётся дневным в `rates`).
- **NG2**: графики/визуализация внутридневных тиков ETH — храним историю, но рисуем только индикатор
  свежести; полноценный intraday-чарт — отдельная задача по сигналу.
- **NG3**: другие монеты кроме ETH (BTC и т.п.) — вне scope; цепочка провайдеров обобщаема, но
  включаем только ETH/EUR.
- **NG4**: алертинг в Telegram при падении всех провайдеров — вне scope (AI-трек/нуджи проектируются
  отдельно); пока видимость через `refresh-rates` + логи cron.
- **NG5**: ретеншн/чистка `rate_ticks` — при 4×/день и 1 монете рост ~1460 строк/год, не проблема;
  чистку отложим (open question OQ1).

## 4. User journeys

### Happy path
1. Cron срабатывает 4×/сутки. Для ETH пробует Binance → (упал) Coinbase → (упал) CoinGecko; первый
   успешный даёт цену. Пишет тик в `rate_ticks` (история) и обновляет дневной курс в `rates` (закрытие дня).
2. Stepan открывает «Инвестиции» в Web Admin → стоимость ETH-портфеля посчитана по **последнему тику**;
   рядом — «курс обновлён N минут назад».
3. Если Binance заблокирован для CF-IP — курс всё равно приходит с Coinbase/CoinGecko; дырки в наполнении нет.

### Edge cases
- **E1**: все три провайдера упали в один слот → тик не пишется, последний известный остаётся; следующий
  слот (≤6ч) наверстает. `refresh-rates.crypto_error` содержит причину.
- **E2**: `rate_ticks` пуст (первый деплой, до первого фетча) → mark-to-market падает назад на дневной
  `rates` (последний доступный курс) — поведение как до SPEC-028, без 0/null-регрессий.
- **E3**: клиентский `today` в часовом поясе опережает/отстаёт от UTC-даты тика → mark-to-market берёт
  тик по `fetched_at` (не по дате), поэтому timezone не влияет на «текущую» стоимость.
- **E4**: историческая трата/доход за прошлую дату → конвертируется по **дневному** курсу той даты
  (как и раньше), тики не влияют на прошлое.
- **E5**: провайдер ответил 200, но мусор/0/отрицательное в цене → провайдер считается упавшим, пробуем
  следующий (как с HTTP-ошибкой).

## 5. Data model

Новая таблица внутридневных тиков. Дневная `rates` **не меняется** (нулевой риск регрессии канонического
слоя конвертации, ADR-014): крипто-фетч продолжает писать в `rates` дневной курс (закрытие дня) для
historical-серий, и **дополнительно** пишет тик.

```sql
-- 0016_rate_ticks.sql
-- Внутридневные тики курса (SPEC-028). Для mark-to-market «сейчас» (по времени фетча).
-- Историческая конвертация (поток по дате) использует дневной rates — здесь не участвует.
CREATE TABLE IF NOT EXISTS rate_ticks (
    base       TEXT NOT NULL,                 -- 'EUR' (фиксированная база)
    quote      TEXT NOT NULL,                 -- 'ETH' (крипта; фиат тики не пишет, NG1)
    rate       REAL NOT NULL,                 -- 1 base = rate × quote (инверсия 1/price, как в rates)
    source     TEXT NOT NULL,                 -- провайдер: 'binance' | 'coinbase' | 'coingecko'
    fetched_at TEXT NOT NULL DEFAULT (datetime('now')),  -- момент фетча (UTC) — авторитет свежести
    PRIMARY KEY (base, quote, fetched_at)
);
CREATE INDEX IF NOT EXISTS idx_rate_ticks_quote_fetched ON rate_ticks(quote, fetched_at DESC);
```

Семантика:
- `rate` — та же инверсия, что и в `rates`: `1 EUR = rate × ETH`, т.е. `rate = 1/price_eur`. `toEurAt`
  и весь существующий слой работают без спец-обработки крипты.
- `fetched_at` — авторитетный признак свежести (а не `date`). Самый свежий тик = `MAX(fetched_at)`.
- `source` — какой провайдер дал котировку (диагностика, доверие).

## 6. API contract

### `POST /v1/admin/refresh-rates` (расширение)
- Auth: Bearer `SYNC_TOKEN` (без изменений).
- Response 200 (добавлены поля):
  ```json
  {
    "ok": true, "saved": 5, "date": "2026-06-09",
    "crypto_saved": 1, "crypto_provider": "coinbase", "crypto_error": null,
    "lido_apr": 2.48, "lido_error": null
  }
  ```
  `crypto_provider` — кто реально отдал курс (null если все упали); `crypto_error` — агрегированная
  ошибка цепочки если `crypto_saved=0`.

### `GET /v1/expenses/rates` (Mini App, расширение getLatestRates)
- Auth: Telegram initData HMAC (без изменений).
- Крипто-quote (ETH) в ответе берётся из **последнего тика** (`rate_ticks`), фиат — из дневного `rates`.
- Добавлено поле `fetched_at` (момент свежести крипты) — опционально для подсказок.

Внутренний контракт `RatesIndex` (canonical, ADR-014):
- `rateAt(quote, date)` становится **tick-aware**: для quote с тиками и `date >= lastDailyDate(quote)`
  (запрос «на сейчас/последнюю дату») возвращает курс из последнего тика; для прошлых дат — дневной
  historical (бинпоиск ≤ date), без изменений. Фиат тиков не имеет → всегда дневной.
- Все mark-to-market точки (net worth «сейчас», стоимость инвест-позиции, goal balance, /accounts EUR)
  уже зовут `…At(.., today)` → автоматически получают свежий тик и остаются **взаимно консистентными**
  (KPI ↔ net-worth-series-tail ↔ /accounts ↔ investments). Точечных правок потребителей не требуется.

## 7. UI / UX

Страница «Инвестиции» (Web Admin): рядом с KPI «Стоимость портфеля» / на карточке позиции — строка
свежести курса: «курс ETH обновлён N мин назад» (из `fetched_at` последнего тика). Состояния:
- свежий (< 6ч) — нейтрально;
- устаревший (≥ 12ч, напр. все провайдеры падают) — приглушённо-предупреждающе (визуальный сигнал, что
  цена могла отстать). Без CRUD, read-only.

## 8. Security

- Провайдеры — **публичные**, без ключей/секретов (Binance/Coinbase/CoinGecko spot). В код/логи секреты
  не попадают.
- Валидация ответа: парсим только число цены, `isFinite && > 0`; иначе провайдер «упал». В SQL уходит
  только `bind` (rate/source/quote — контролируемые значения).
- `refresh-rates` остаётся за `SYNC_TOKEN` (мутация). `getLatestRates` — за initData HMAC.
- В логи cron — только имя провайдера и текст ошибки (HTTP-статус), без тел ответов.

## 9. Acceptance criteria

- [ ] AC1: cron сконфигурирован на 4×/сутки (`0 */6 * * *`).
- [ ] AC2: при недоступности Binance крипто-фетч берёт курс с Coinbase, при недоступности обоих —
  с CoinGecko (unit-тест с мок-fetch, fallback-цепочка).
- [ ] AC3: если все провайдеры упали — `saveCryptoTick` не пишет, последний курс остаётся, `refresh-rates`
  возвращает `crypto_saved=0` + непустой `crypto_error`.
- [ ] AC4: каждый успешный фетч пишет строку в `rate_ticks` (история) и обновляет дневной `rates`.
- [ ] AC5: `RatesIndex.rateAt('ETH', today)` возвращает курс **последнего тика**; `rateAt('ETH', pastDate)`
  возвращает дневной historical (unit-тест tick-aware).
- [ ] AC6: стоимость ETH в `/investments`, `/dashboard` (netNow + tail net-worth-series) и `/accounts`
  совпадает (одна цена — последний тик; тест консистентности).
- [ ] AC7: историческая конвертация трат/доходов/cost basis (поток по дате) не изменилась (регресс-тесты
  budgets/dashboard/investments зелёные).
- [ ] AC8: `refresh-rates` на проде возвращает `crypto_saved=1` и `crypto_provider` ∈ {binance,coinbase,coingecko};
  ETH-курс на проде свежий (fetched_at в пределах часа после запуска).
- [ ] AC9: страница «Инвестиции» показывает время последнего фетча курса.

  оба падают → CoinGecko; все падают → throw); tick-aware `rateAt` (today→tick, past→daily, EUR=1, нет
  тика→daily fallback). `investments.test.ts`/`dashboard.test.ts` — консистентность стоимости ETH по

- **Regression**: `npm run typecheck` + весь `vitest run` (budgets/dashboard/ledger/effective-balance —

- **Prod smoke**: `refresh-rates` → проверить `crypto_provider`/`crypto_saved`; `rate_ticks` пополнилась;
  ETH-стоимость портфеля на проде свежая.
- **Admin SPA**: Playwright/скриншот light/dark страницы инвестиций с индикатором свежести (правило
  frontend-test-locally-before-deploy).

## 11. Risks & open questions

- **R1**: fallback-провайдеры тоже могут гео-блокировать CF-IP. Митигация: 3 независимых провайдера
  (Binance/Coinbase/CoinGecko — разные политики); Coinbase/CoinGecko исторически дружелюбны к serverless.
  После деплоя `crypto_provider` покажет, кто реально проходит с CF-IP.
- **R2**: CoinGecko free-tier rate-limit (~10–30 req/min) — при 4×/день и роли последнего fallback
  несущественно.
- **R3**: рост `rate_ticks` (~1460 строк/год на ETH) — приемлемо; чистка отложена.
- **OQ1**: нужна ли ретенция/прореживание `rate_ticks` со временем? (пока нет, пересмотреть если >10k строк).

## 12. Out of scope для review

- Дублирование записи крипто-курса (дневной `rates` + `rate_ticks`) — сознательно: разные цели
  (historical daily series vs intraday mark-to-market), изоляция от канонического слоя.
- Семантика «date == lastDailyDate → tick» для редкой ETH-операции-потока за сегодня (берётся свежий
  тик вместо дневного закрытия) — расхождение копеечное, осознанно.
