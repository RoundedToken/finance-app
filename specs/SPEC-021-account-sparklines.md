# Sparklines баланса на карточках /accounts

> Пост-MVP Фаза 2, пункт **2.1** (`docs/post-mvp-roadmap.md`), хвост Stage 5b. V/E = **H/S**.

## 1. Context & Problem

На `/accounts` каждое ведро показывает текущий баланс (native + ≈EUR + drift + events), но **не видно «куда движется»** — растёт счёт или тает. Чтобы понять тренд, надо открыть дашборд. При этом помесячный `balanceAt(bucket, asOf)` по каждому ведру **уже считается** на worker в `dashboard.ts` (`net_worth_series[].by_bucket`), но в EUR и схлопнут в агрегаты — нативный ряд по ведру наружу не отдаётся. Дёшево добавить мини-тренд прямо под балансом карточки.

## 2. Goals

- **G1**: Под балансом каждого ведра на `/accounts` — мини-спарклайн помесячного **нативного** баланса за окно дашборда (дефолт 12 мес), в цвете ведра. Последняя точка совпадает с большим числом карточки.
- **G2**: Под KPI-карточкой «Net worth» (сводка) — спарклайн помесячного `total_eur` (EUR — корректно для net worth), спокойным нейтральным цветом.
- **G3**: Ноль дублирования математики: нативный ряд = тот же `balanceAt`, что уже считает dashboard; Sparkline-компонент извлечён из `DashboardPage` в общий `components/Sparkline.tsx` и переиспользован (DRY).

## 3. Non-Goals

- **NG1**: Никаких осей, тултипов, drill-down на спарклайнах карточек — это «искра тренда», не график. Полный график баланса по ведру — отдельный Stage 5b.
- **NG2**: Не меняем денежную математику, не добавляем миграций, не добавляем новых endpoint'ов. `by_bucket_native` — производное поле поверх уже считающегося `balanceAt`.
- **NG3**: Не трогаем Mini App (правило 11 — аналитика только в Admin).
- **NG4**: Спарклайн в EUR-эквиваленте на ведре — **отвергнут** (owner-решение 2026-05-31): под native-числом EUR-линия смешивала бы движение денег и колебания курса (двусмысленно для одной карточки) + missing-rate месяц давал бы фейковый провал в 0.

## 4. User journeys

### Happy path
1. Открываю `/accounts`. Под балансом «EUR · банк = 1 040 €» (или «RSD · нал = 120 000 RSD») вижу тонкую линию-тренд за последние ~12 мес в цвете ведра.
2. Линия идёт вверх → ведро наполняется; вниз → тает. Мгновенно, без открытия дашборда.
3. Под главным «Net worth» в сводке — такая же искра по EUR-итогу: куда движется капитал в целом.

### Edge cases
- **E1 — ведро без вариации** (пустое / баланс не менялся за окно): спарклайн **скрыт** (плоская линия не несёт сигнала; число баланса всё равно на карточке). Правило: рисуем только если `max(series) !== min(series)` и точек ≥ 2.
- **E2 — dashboard ещё грузится / ошибка**: спарклайн не рисуется, ядро карточек рендерится как раньше (graceful degradation, отдельный query — не блокирует `/accounts`).
- **E3 — < 2 месяцев истории**: спарклайн скрыт (нужно ≥ 2 точек).
- **E4 — missing-rate в каком-то месяце**: для нативного ряда не актуально (нет конверсии). Для EUR-сводки (G2) — пре-существующее поведение `total_eur` (точка может занижаться), не регрессия этой фичи.

## 5. Data model

Без изменений D1. Никаких миграций.

## 6. API contract

### `GET /v1/web/dashboard` (расширение ответа, без изменения контракта запроса)
- Auth: JWT (Bearer) — без изменений.
- В каждый элемент `net_worth_series[]` добавляется поле:
  ```jsonc
  {
    "month": "2026-05",
    "total_eur": 23456.0,
    "by_bucket": { "rsd-bank": 1040.0 },          // EUR (как было)
    "by_bucket_native": { "rsd-bank": 120000.0 }, // NEW: native-валюта ведра (balanceAt до конверсии)
    "by_form": { ... }, "by_currency": { ... }
  }
  ```
- `by_bucket_native[id]` = `balanceAt(id, T)` (тот же, что уже считается строкой выше конверсии), округлён `r2`. Для текущего месяца `T = today` → совпадает с `effective_balance` из `/v1/web/accounts`.
- `/v1/web/accounts` — **без изменений** (ряд берём из dashboard, чтобы не дублировать агрегатор).

## 7. UI / UX

```
┌─ Карточка ведра ───────────────────────────┐
│ 🟦 EUR · банк                          ↗   │
│                                            │
│ 120 000 RSD            ← большое число     │
│ ≈ 1 040 € · baseline … · +X от событий …   │
│ ╭───────────────────────────────────────╮ │  ← Sparkline (native, цвет ведра, h≈28)
│ │      ╱╲      ╱                          │ │     скрыт если плоский/нет данных
│ ╰───────────────────────────────────────╯ │
└────────────────────────────────────────────┘
```

- Спарклайн: ECharts line, `smooth`, `showSymbol:false`, без осей/тултипа, лёгкая area-заливка (как на дашборде). Высота ~28px на карточке ведра, ~34px на сводке.
- Цвет ведра: `acc.color` (как фон/цвет иконки), fallback `#9ca3af`. Сплошная линия (не пунктир — пунктир зарезервирован за forecast).
- Net worth сводка (G2): нейтральный/primary тон (спокойный, не сигнальный — линия = тренд капитала, не «хорошо/плохо»).
- Состояния loading/error дашборда — спарклайн просто отсутствует, ядро карточек не страдает.

## 8. Security

- Read-only, тот же JWT-gate на `/v1/web/dashboard`. Новых данных наружу нет — `by_bucket_native` это та же финансовая позиция, что и `effective_balance` (уже отдаётся на `/accounts`). PII не добавляется. В логи ничего нового.

## 9. Acceptance criteria

- [x] AC1: Worker `/v1/web/dashboard` отдаёт `by_bucket_native` в каждом `net_worth_series[]`; для текущего месяца значение по ведру == `effective_balance` того же ведра из `/v1/web/accounts` (та же `balanceAt`/`reconstructBalance`; tsc зелёный, искры рендерятся из поля).
- [x] AC2: На `/accounts` под балансом каждого ведра с вариацией баланса виден спарклайн в цвете ведра; последняя точка визуально соответствует большому числу (скрин light/dark).
- [x] AC3: Ведро без вариации/без данных — спарклайн отсутствует (eur-cash плоско, try-cash пусто → без искры), карточка рендерится корректно.
- [x] AC4: Под сводным «Net worth» виден спарклайн `total_eur` нейтральным тоном (G2).
- [x] AC5: При loading/error дашборда `/accounts` рендерит карточки без спарклайнов (отдельный query, `dash?.…` → null), без белого экрана.
- [x] AC6: `Sparkline` живёт в `components/Sparkline.tsx`, переиспользован и дашбордом, и `/accounts`; дашбордные KPI-спарклайны не изменились визуально (regression-скрин).
- [x] AC7: `tsc` зелёный (worker + admin); 51 vitest зелёные.

## 10. Test plan

- **Worker**: vitest/смоук — `getDashboard` возвращает `by_bucket_native`, значение == native `balanceAt`; проверка совпадения с `getEffectiveBalance(today)` (математика уже покрыта `reconstructBalance`-тестами — новой математики нет).
- **Admin SPA**: Playwright (`test_admin_ui.py`) — `/accounts` light/dark, мок `/v1/web/dashboard` с `by_bucket_native`, скриншоты карточек со спарклайнами + кейс «нет вариации» (скрыт). Дашборд — regression-скриншот KPI-спарклайнов.
- **Regression**: DashboardPage (извлечение компонента не сломало KPI-спарки), AccountsPage core (баланс/drift/summary).

## 11. Risks & open questions

- **R1**: Coupling `/accounts` → `/dashboard` payload. Принято осознанно: тренд-данные имеют один дом (dashboard-агрегатор), потребляются в двух местах (графики дашборда + искры accounts) — это корректная факторизация, а не лишняя связь. Деградация мягкая (искра — enhancement, отдельный query, не блокирует ядро `/accounts`).
- **R2**: Лишний `/dashboard`-запрос. **Уточнено после Phase 3:** кэш `/` → `/accounts` фактически **не** шарится — DashboardPage шлёт `useDashboard({from,to})` (queryKey `['dashboard','<from>','<to>']`), AccountsPage — `useDashboard()` (queryKey `['dashboard',null,null]`); это разные записи кэша. Значит первый заход на `/accounts` всегда триггерит один батч-запрос дашборда (сервер отдаёт дефолтное 12-мес окно — те же данные). Кэш реально шарится только `/accounts` → `/accounts` в пределах staleTime 30s. Унификацию ключа не делаем: связала бы AccountsPage с пресет-логикой окна дашборда ради экономии одного батч-запроса (~11 D1-чтений) у single-user — не стоит. Альтернатива (нативный ряд прямо в `/v1/web/accounts`) дублировала бы агрегатор (анти-цель 2.7).
- **R3**: `by_bucket_native` для USDT округляется `r2` — для 34px-тренда достаточно; точные 8 знаков крипты не нужны.

## 12. Out of scope для review

- Двойной `loadRatesIndex` / общий aggregator dashboard↔accounts (тех-долг 2.7) — не в этой итерации.
- EUR-vs-native в сводке: net worth по смыслу EUR — здесь это не «несогласованность».
