# Робастный месячный темп: медиана за 6 месяцев вместо среднего за 3

## 1. Context & Problem

Все «темповые» метрики дашборда (KPI «Доход/мес», «Траты/мес», «Норма сбережений», runway, пунктир-прогноз net worth, прогноз достижения целей) считаются от **простого среднего за 3 полных месяца** (`windowSums`, `dashboard.ts`, WIN=3). Среднее не устойчиво к выбросам: одна разовая покупка техники ~4500 € в мае 2026 сделала темп отрицательным (−546 €/мес в линзе «Свободные») и на три месяца «сломала» прогноз (пунктир к октябрю уводит свободные деньги к 3165 €) и норму сбережений (−15 %) — при том, что чистый июнь в профиците +2069 €. Owner-решение 2026-07-01: **медиана по 6 полным месяцам, применить ко всем потребителям темпа**.

Вторая (смежная) проблема — дашборд противоречит сам себе в линзе «Свободные»: KPI-спарклайн Net worth вычитает `invested_eur` **по-точечно** (`DashboardPage.tsx:192`), а большой график — **текущей константой** из всех исторических точек (`DashboardPage.tsx:332`). В июне (покупка ETH ~3370 €) спарклайн идёт вниз, а большая линия вверх — из одних и тех же данных.

## 2. Goals

- G1: Единая функция «типичный месячный темп»: **медиана** помесячных `income`, `income_free`, `burn` (EUR по курсу даты операции) за последние **6 полных календарных месяцев** (текущий месяц исключён). Все потребители — KPI burn/income, `savings_rate(_free)`, `runway(_total)`, `projectRate` пунктира, `GoalsForecastSection` — читают её.
- G2: Δ-бейджи KPI сравнивают текущее 6-мес окно с предыдущим 6-мес окном (та же медиана); `prev_net_worth_eur` считается на конец месяца −6 (вместо −3). Одно окно = один сигнал во всех элементах карточки (принцип «KPI = один сигнал»).
- G3: Большой график net worth в линзе «Свободные» вычитает `invested_eur` **по-точечно** из каждой точки серии (как KPI-спарклайн); `targeted_eur` остаётся константой (историю goal-balance не реконструируем — существующее осознанное ограничение).
- G4: UI честно подписывает семантику: KPI-карточки — «медиана за 6 мес», серия прогноза — «Прогноз (медиана 6 мес)».

## 3. Non-Goals

- NG1: Пометка трат «разовая» (one-off флаг) и их исключение из баров/donut — отдельная будущая фича.
- NG2: Holt/SPC/ML-прогнозирование сальдо — не в этой итерации (RBAR-математика SPEC-023 остаётся только в бюджетах).
- NG3: Бары «Доходы vs Расходы», donut и `cashflow_series` не меняются — факт остаётся фактом.
- NG4: Реконструкция исторического `targeted` (goal balance по месяцам) — не делаем.
- NG5: Учёт расходов без `account_id` в реконструкции балансов (`dashboard.ts:136`) — известная слабость, вынесена в tech debt, не здесь.

## 4. User journeys

### Happy path

1. Stepan открывает дашборд в линзе «Свободные» после разовой крупной траты в прошлом месяце.
2. KPI «Траты/мес» показывает типичный (медианный) уровень трат — выброс не влияет; «Норма сбережений» положительная, соответствует обычным месяцам.
3. Пунктир-прогноз net worth идёт по типичному темпу (+~880 €/мес на данных июня 2026), а не в минус.
4. Спарклайн в карточке Net worth и большая линия графика имеют один и тот же наклон за последний месяц.

### Edge cases

- E1: История < 6 полных месяцев → медиана по фактически покрытым месяцам (M2-механизм `monthsCovered` сохраняется). 1 месяц → его значение; 0 месяцев → 0, `savings_rate = null`, прогноз плоский.
- E2: Чётное число значений → среднее двух центральных (классическая медиана).
- E3: `prev`-окно покрыто частично → медиана по покрытым; не покрыто вовсе → prev-значения = 0, Δ-бейджи ведут себя как сейчас при пустой истории.
- E4: Все 6 месяцев «выбросные» (например, переезд) — медиана честно отражает новую норму; ускорять адаптацию не пытаемся (осознанный trade-off выбора «медиана»).

## 5. Data model

Без изменений D1. Медиана — on-read в `getDashboard` из уже загружаемых `expenses`/`incomes`.

## 6. API contract

Shape `GET /v1/web/dashboard` **не меняется** — те же поля `kpi.*`. Меняется семантика:

- `monthly_burn_eur`, `monthly_income_eur`, `monthly_income_free_eur` — медианы помесячных сумм за 6 полных месяцев (было: среднее за 3).
- `prev_monthly_*` — те же медианы за предыдущее 6-мес окно (shift=6).
- `savings_rate(_free)`, `runway_months(_total)` — производные от медианных значений.
- `prev_net_worth_eur`, `prev_free_net_worth_eur` — на конец месяца −6.

Потребители полей: Web Admin (KPI, пунктир, GoalsForecastSection) и **coach.ts** (SPEC-040: нудж `runway_low` читает `runway_months`/`free_net_worth_eur` из того же `getDashboard` — теперь от медианного burn, что в духе фичи: нудж не сработает ложно от разового выброса). Mini App статистика — свой endpoint, не задета.

## 7. UI / UX

- KPI-карточки «Доход», «Траты», «Норма сбережений»: подпись `sub` — «медиана за 6 мес» (вместо текущей «в среднем за 3 мес», если есть).
- `NetWorthChart`: имя серии пунктира — «Прогноз (медиана 6 мес)»; в линзе «Свободные» hist-серия = `max(0, total_eur(point) − targeted_eur − invested_eur(point))` — по-точечный `invested`.
- Спарклайн KPI Net worth — без изменений (уже per-point).

## 8. Security

Мутаций нет, auth не меняется (JWT-guard `/v1/web/*` как был). Новых входных данных нет.

## 9. Acceptance criteria

- [ ] AC1: `median([])=0`, `median([x])=x`, нечётное → центральный, чётное → среднее двух центральных (unit-тест).
- [ ] AC2: На данных прода июля 2026 (`today=2026-07-01`): окно = янв–июн 2026; `monthly_burn_eur` = медиана 6 месячных burn — майский выброс (7627 €) не является медианой; `projectRate` в линзе «Свободные» > 0.
- [ ] AC3: `savings_rate` и `runway` считаются от медианных значений; «Норма сбережений» (free) на данных прода положительная.
- [ ] AC4: Δ-бейджи: prev-окно = 6 мес до текущего окна; `prev_net_worth_eur` — на `endOfMonth(curMonth − 6)`.
- [ ] AC5: В линзе «Свободные» большая линия net worth и KPI-спарклайн имеют одинаковый знак изменения за последний завершённый месяц (per-point invested в обоих).
- [ ] AC6: Подписи UI обновлены: KPI sub «медиана за 6 мес», легенда пунктира «Прогноз (медиана 6 мес)».
- [ ] AC7: История < 6 месяцев не ломает расчёт (медиана по покрытым месяцам, нет деления на 0).

## 10. Test plan

- **Worker**: vitest — unit на `median()` и `windowSums`-замену (синтетические месяцы с выбросом: медиана игнорирует выброс, среднее — нет); существующие тесты dashboard не регрессируют.
- **Admin SPA**: `local/scripts/test_admin_ui.py` (Playwright, mock JWT + mock `/v1/**`) — скриншоты light/dark дашборда, проверка подписей и пунктира.
- **Prod smoke**: после деплоя — `GET /v1/web/dashboard` (JWT), сверить `kpi.monthly_burn_eur` с рассчитанной вручную медианой (данные этой сессии).
- **Regression**: `GoalsForecastSection` (прогноз целей использует те же KPI), Mini App не задет.

## 11. Risks & open questions

- R1: Пользователь привыкнет к «типичной» норме сбережений и не заметит реального устойчивого роста трат первые ~3 месяца (лаг медианы). Принято осознанно (выбор алгоритма 2026-07-01).
- R2: Семантика Δ-бейджей меняется с «3 мес vs 3 мес» на «6 vs 6» — более инертная. Принято (G2, единое окно).
- R3: Sparse-потоки медиана обостряет: если ≥ половины покрытых месяцев окна без трат/дохода, медиана = 0 → Runway «∞» / savings_rate null («—»). Среднее деградировало плавно. Для ежедневного трекинга не проявляется; проявится у нового пользователя с редкими операциями. Осознанный trade-off (см. E1/E4).

## 12. Out of scope для review

- Учёт расходов без `account_id` в `balanceAt` (NG5, tech debt).
- Точность исторического `targeted` в линзе «Свободные» (NG4).
