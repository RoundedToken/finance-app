# SPEC-051 · Realized P&L + оценка стейкинг-дохода (кандидат, BP-03+BP-06 аудита 2026-07)

> **Draft / SPEC-кандидат.** Одна инвест-итерация, желательно ДО первой продажи ETH
> (после неё P&L-история смешивается необратимо и реконструируется только из дампов).

## 1. Context & Problem

- **BP-03 (P2):** продажа ETH→USDT — обычный `exchange` (NG3 SPEC-026/ADR-018);
  realized P&L нигде не фиксируется. `/investments` показывает только unrealized
  (стоимость − WAC-basis). Для налоговой декларации cost basis по каждой продаже
  придётся собирать вручную.
- **BP-06 (P3):** доход стейкинга (`qty(today) − net_bought_qty`) оценивается по
  сегодняшнему курсу целиком; момент получения награды не фиксируется — накопленный
  доход «дышит» с курсом, а большинство налоговых режимов оценивают стейкинг-награды
  по FMV на дату получения.

## 2. Design (эскиз из аудита)

- **Realized P&L**: on-read линза в `investments.ts` (без миграции): хронологический
  проход exchange-истории инвест-ведра с running-WAC; на каждой продаже
  `realized_eur = to_amount→EUR (поток, по дате) − WAC × qty_sold`. UI `/investments`:
  строка «Realized P&L» (накопленный итог) + список продаж. Лоты/FIFO не вводить —
  average-метод (ACB) достаточен и легитимен для single-user.
- **Стейкинг-доход**: прирост qty фиксировать по курсу даты снапшота, в котором он
  впервые виден (аппроксимация даты получения) — накопленный доход перестаёт дышать
  задним числом; текущая переоценка остаётся отдельной строкой.

## 3. Open Questions (owner)

- OQ1: показывать realized P&L в EUR only или и в native (USDT-выручке)?
- OQ2: стейкинг-фиксация — по дате снапшота (дёшево, эскиз выше) или точнее
  (ежедневный аккруал по APR — сложнее, но ровнее)?
- OQ3: нужна ли сводная строка «итого P&L = realized + unrealized + staking» на карточке.

## 4. Оценка

M (обе части — on-read, без миграций; UI — таблица продаж + 2 строки KPI).
